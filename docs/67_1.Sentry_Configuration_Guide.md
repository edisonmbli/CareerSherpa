# Sentry 内部配置规范（一页版）

## 1. 目标

Sentry 接入的验收标准只有 4 条：

1. Web 错误进入 **Web 项目**
2. Worker 错误进入 **Worker 项目**
3. 事件带有正确的 **release**
4. `bug feedback` 能关联到 Sentry，并出现 **Open Sentry**

只要这 4 条成立，就视为接入完成。 

---

## 2. 项目划分规范

必须拆成两个 Sentry 项目，不允许混用：

* `career-shaper-web`
* `career-shaper-worker`

规则：

* `NEXT_PUBLIC_SENTRY_DSN` 只给 **Web**
* `SENTRY_DSN` 只给 **Worker**

不要把 Web 和 Worker 放进同一个项目，否则会导致 issue 混杂、release 难追踪、source maps 排查变差。 

---

## 3. 后台准备项

Sentry 后台需要准备：

* Web 项目的 DSN
* Worker 项目的 DSN
* `Auth Token`
* `Org slug`
* `Web project slug`
* `Worker project slug`

`Auth Token` 主要用于：

* 构建时上传 source maps
* feedback 中 `Open Sentry` 的链接解析

建议最少权限：

* `Project: Read`
* `Project: Write`
* `Release: Admin`
* `Org: Read`  

---

## 4. 环境变量规范

### 本地开发

本地也必须分开配置 Web 和 Worker，否则无法验证事件是否进对项目。

**Next.js / Web**

```bash
NEXT_PUBLIC_SENTRY_DSN=<web dsn>

SENTRY_AUTH_TOKEN=sntrys_...
SENTRY_ORG=<org-slug>
SENTRY_PROJECT=career-shaper-web
SENTRY_WORKER_PROJECT=career-shaper-worker

SENTRY_ENVIRONMENT=development
SENTRY_RELEASE=local-dev

NEXT_PUBLIC_SENTRY_ENABLED=true
SENTRY_ENABLED=true
```

**Worker**

```bash
SENTRY_DSN=<worker dsn>

SENTRY_AUTH_TOKEN=sntrys_...
SENTRY_ORG=<org-slug>
SENTRY_WORKER_PROJECT=career-shaper-worker

SENTRY_ENVIRONMENT=development
SENTRY_RELEASE=local-dev

SENTRY_ENABLED=true
```

### 生产环境

**Web（Vercel）**

```bash
NEXT_PUBLIC_SENTRY_DSN=<web dsn>

SENTRY_AUTH_TOKEN=sntrys_...
SENTRY_ORG=<org-slug>
SENTRY_PROJECT=career-shaper-web

SENTRY_ENVIRONMENT=production
NEXT_PUBLIC_SENTRY_ENABLED=true
SENTRY_ENABLED=true
```

**Worker**

```bash
SENTRY_DSN=<worker dsn>

SENTRY_AUTH_TOKEN=sntrys_...
SENTRY_ORG=<org-slug>
SENTRY_WORKER_PROJECT=career-shaper-worker

SENTRY_ENVIRONMENT=production
SENTRY_RELEASE=<same-git-sha-as-web>

SENTRY_ENABLED=true
```

---

## 5. Release 规范

`release` 的定义：

> 当前这次代码发布版本的唯一标识

推荐统一使用：

* **git commit SHA**

例如：

```bash
SENTRY_RELEASE=abc1234
```

用途只有三个：

* 标识错误属于哪一版代码
* 让 source maps 正确匹配
* 便于按版本追踪问题

统一要求：

* 本地：`SENTRY_RELEASE=local-dev`
* 生产：**Web 和 Worker 必须使用同一个 release**

否则 source maps 和事件可能对不上。  

---

## 6. Release 注入规则

不要把下面这种动态命令直接写进 `.env`，并假设它会自动执行：

```bash
SENTRY_RELEASE=$(git rev-parse --short HEAD)
```

规范要求：

* `.env` 里只放**最终值**
* 动态生成应由 **CI/CD / 部署脚本 / 启动脚本** 完成

### 推荐做法

**Web（Vercel）**

* 可优先使用平台自动注入的 commit SHA
* 当前代码如果已兼容 `SENTRY_RELEASE` / `VERCEL_GIT_COMMIT_SHA` / `NEXT_PUBLIC_VERCEL_GIT_COMMIT_SHA`，则 Web 可不手动写死 release。 

**Worker**

* 在部署脚本里注入和 Web 相同的 git SHA
* 不允许手工随意改成别的值

---

## 7. Environment 命名规范

统一使用以下 3 个值：

* `development`
* `preview`
* `production`

不要自定义其他命名，避免后续过滤、告警、统计口径不一致。

---

## 8. 验收 Checklist

### 本地联调

完成配置后，按顺序验证：

1. 重启开发环境
2. 打开 `/zh/sentry`
3. 测试：

   * client exception
   * server route exception
   * edge route exception
4. 启动 worker 后访问 `/debug/sentry`

确认结果：

* Web 事件进入 `career-shaper-web`
* Worker 事件进入 `career-shaper-worker`
* `environment` 正确
* `release` 正确 

### Feedback 联动

再做一轮验证：

1. 先制造一次前端异常
2. 在同一页面提交一条 `bug` feedback
3. 到 Slack 看是否出现 `Open Sentry`
4. 再提交 `feature` 或 `confusion`
5. 确认这两类 **不出现** `Open Sentry`  

---

## 9. Source Maps 检查项

生产构建后，到 Sentry 检查：

1. Release 是否存在
2. Event 是否带有 `release`
3. 堆栈是否映射回源码，而不是压缩产物

如果映射失败，优先排查：

* `SENTRY_AUTH_TOKEN`
* `SENTRY_ORG`
* `SENTRY_PROJECT`
* `SENTRY_RELEASE`
* 构建流程是否真的执行了 source maps 上传 

---

## 10. 禁止事项 / 常见错误

### 禁止 1：Web / Worker 共用一个 DSN

错误做法：

* `NEXT_PUBLIC_SENTRY_DSN` 和 `SENTRY_DSN` 填同一个值

正确做法：

* 前者给 Web
* 后者给 Worker 

### 禁止 2：生产环境 release 不一致

错误做法：

* Web 和 Worker 使用不同版本号

正确做法：

* 两端必须共用同一个 `SENTRY_RELEASE`  

### 禁止 3：只看 SDK 是否初始化

验收不看“SDK 是否装上”，只看 4 个最终结果是否成立。

---

## 11. 最终口径

Sentry 接入完成的统一定义：

* 项目分离正确
* DSN 使用正确
* release 统一正确
* source maps 可用
* feedback → `Open Sentry` 联动正常

# Sentry 整合配置精炼版

## 1. 目标与总体原则

这次整合的本质，不是“把 SDK 初始化上了”就算完成，而是确保下面四件事成立：

1. **Web 错误进入 Web 项目**
2. **Worker 错误进入 Worker 项目**
3. **事件带着正确的 release**
4. **bug feedback 能关联到 Sentry，并出现 `Open Sentry`**

只要这四项成立，Sentry 这轮才算真正打通。

---

## 2. 项目划分：Web 和 Worker 必须分开

应在 Sentry 中建立两个独立项目：

* `career-shaper-web`
* `career-shaper-worker`

不要把 Web 和 Worker 混到同一个 Sentry 项目里。分开后，问题归属、版本追踪、source maps 排查都会更清晰。

对应关系也要固定：

* `NEXT_PUBLIC_SENTRY_DSN` → 给 **Web**
* `SENTRY_DSN` → 给 **Worker**

这两个 DSN 不是同一个值，不能混用。

---

## 3. 需要准备的后台信息

在 Sentry 后台至少准备这些内容：

* Web 项目的 DSN
* Worker 项目的 DSN
* `Auth Token`
* `Org slug`
* `Web project slug`
* `Worker project slug`

其中 `Auth Token` 主要用于两类事情：

* 构建时上传 source maps
* feedback 中的 `Open Sentry` 链接解析

建议权限最少包含：

* `Project: Read`
* `Project: Write`
* `Release: Admin`
* `Org: Read` 

---

## 4. 环境变量的核心规则

### 本地开发

本地也要区分 Web 和 Worker，因为你需要验证事件是否进对项目。

建议最小配置思路如下：

**Web 侧**

* `NEXT_PUBLIC_SENTRY_DSN`
* `SENTRY_AUTH_TOKEN`
* `SENTRY_ORG`
* `SENTRY_PROJECT=career-shaper-web`
* `SENTRY_WORKER_PROJECT=career-shaper-worker`
* `SENTRY_ENVIRONMENT=development`
* `SENTRY_RELEASE=local-dev`
* `NEXT_PUBLIC_SENTRY_ENABLED=true`
* `SENTRY_ENABLED=true`

**Worker 侧**

* `SENTRY_DSN`
* `SENTRY_AUTH_TOKEN`
* `SENTRY_ORG`
* `SENTRY_WORKER_PROJECT=career-shaper-worker`
* `SENTRY_ENVIRONMENT=development`
* `SENTRY_RELEASE=local-dev`
* `SENTRY_ENABLED=true` 

### 生产环境

生产环境的关键不是“变量多不多”，而是两点：

* Web / Worker 分别用各自 DSN
* Web / Worker 使用**同一个 `SENTRY_RELEASE`**

Web 在 Vercel 上可以直接利用平台提供的 commit 信息；Worker 则通常需要在部署脚本里注入同一个 release。

---

## 5. `release` 的正确理解

`release` 可以直接理解成：

> **这次代码发布版本的唯一标识**

最推荐的值是 **git commit SHA**。例如：

* `abc1234`

它的作用有三件：

1. 标识错误属于哪一版代码
2. 让 source maps 正确匹配
3. 方便在 Sentry 里按版本回溯问题

因此规则很简单：

* **本地**：用 `local-dev` 就够
* **生产**：Web 和 Worker 用同一个 git commit SHA 

---

## 6. `release` 怎么写：不要在 `.env` 里写命令

不要在 `.env` 里写这种内容并指望它自动执行：

```bash
SENTRY_RELEASE=$(git rev-parse --short HEAD)
```

`.env` 一般只接受**静态字符串**，不会自动执行 shell 命令。正确做法是：

* `.env` 里写最终结果值，例如 `SENTRY_RELEASE=abc1234`
* 或者在 **CI / 部署脚本 / 启动脚本** 里先算出这个值，再注入运行环境

结论就是：

* `.env` 写结果
* 脚本负责生成结果 

---

## 7. 生产环境下的推荐做法

### Web（Vercel）

如果代码已经支持下面这类回退逻辑：

* `SENTRY_RELEASE`
* 否则 `VERCEL_GIT_COMMIT_SHA`
* 否则 `NEXT_PUBLIC_VERCEL_GIT_COMMIT_SHA`

那么 Web 这边可以先不手动写死 `SENTRY_RELEASE`，直接使用 Vercel 自动注入的 commit SHA。

### Worker

Worker 没有 Vercel 那样的自动变量时，应在部署流程里注入：

```bash
SENTRY_RELEASE=<与本次 Web 发布一致的 git SHA>
```

最佳实践是：

* 在部署前先拿到当前 commit SHA
* Web 与 Worker 都使用同一个值

这样可以避免 Web/Worker release 不一致，导致 source maps 或问题归属错位。

---

## 8. 最小成本的发布流程

如果当前还没有完整 CI，可以用最小成本方案：

1. 发布前执行 `git rev-parse --short HEAD`
2. 得到这次发布的统一版本号，比如 `abc1234`
3. Web 正常发到 Vercel
4. Worker 部署前把 `SENTRY_RELEASE=abc1234` 写入环境
5. 重启 Worker

这样能保证：

* Web / Worker release 一致
* 错误和 source maps 能稳定对齐
* 手工成本相对最低 

---

## 9. 推荐固定的 environment 命名

环境名建议固定，不要自由发挥：

* 本地：`development`
* 预发：`preview`
* 生产：`production`

这样后续过滤、告警、统计都会简单很多。

生产构建完成后，需要在 Sentry 里确认：

1. Release 是否存在
2. Event 是否带 `release`
3. 堆栈是否能映射回源码，而不是压缩后的产物

如果映射失败，优先排查：

* `SENTRY_AUTH_TOKEN`

## 12. 这件事最容易踩的坑

最常见的坑其实就两个：

### 坑 1：把 Web 和 Worker 的 DSN 混了

正确做法：

* `NEXT_PUBLIC_SENTRY_DSN` 只给 Web
* `SENTRY_DSN` 只给 Worker

### 坑 2：生产环境的 release 不一致

如果 Web 和 Worker 的 `SENTRY_RELEASE` 不一致，就很容易出现：

* source maps 对不上
* 错误归属混乱
* 排查体验变差

# 一句话结论

这套配置最核心的干货只有三条：

* **Web 和 Worker 必须分开上报**
* **生产环境必须共享同一个 `SENTRY_RELEASE`**
* **验收时只盯 4 个结果：项目归属、release、environment、`Open Sentry` 联动**

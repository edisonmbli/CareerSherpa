# Analytics / Outbox Cron Cleanup Guide (v3)

本文档描述 v3 版本下与 Analytics 相关的定时任务设计、保留策略和上线检查项。

## 1. 为什么 v3 仍需要 DB 清理

即使 v3 已接入 PostHog，后台仍然将 analytics 事件写入 `analytics_events`，并通过 `analytics_outbox` 异步同步到 PostHog。

因此必须保留清理任务，否则 Neon Free Tier 很容易被日志表与 outbox 表长期增长拖满。

## 2. 当前 cron 任务（v3）

文件：`vercel.json`

```json
{
  "crons": [
    {
      "path": "/api/cron/flush-analytics-outbox",
      "schedule": "2,12,22,32,42,52 * * * *"
    },
    {
      "path": "/api/cron/cleanup-analytics",
      "schedule": "0 0 * * *"
    },
    {
      "path": "/api/cron/cleanup-analytics-outbox",
      "schedule": "20 0 * * *"
    },
    {
      "path": "/api/cron/cleanup-share-avatars",
      "schedule": "40 0 * * *"
    }
  ]
}
```

说明：
1. `flush-analytics-outbox`：高频小批量同步到 PostHog。
2. `cleanup-analytics`：清理过期 `analytics_events`。
3. `cleanup-analytics-outbox`：清理已导出且超过保留期的 outbox 记录。
4. `cleanup-share-avatars`：清理过期分享头像（对象存储 + DB 引用）。

## 3. 保留策略

### 3.1 analytics_events
文件：`lib/dal/analyticsEvent.ts`

- `SYSTEM`：30 天
- `BUSINESS`：90 天
- `SECURITY`：90 天

### 3.2 analytics_outbox
文件：`lib/dal/analyticsEvent.ts`

- 仅删除 `exported_at < now - retentionDays`
- 默认保留期：7 天
- 环境变量可调：`ANALYTICS_OUTBOX_RETENTION_DAYS`

## 4. 安全策略（重要）

所有 cron route 使用统一鉴权：`lib/cron/auth.ts`

规则：
1. 生产环境 (`NODE_ENV=production`) 必须配置 `CRON_SECRET`，否则返回 500（配置错误，不执行任务）。
2. 当 `CRON_SECRET` 存在时，必须校验 `Authorization: Bearer <CRON_SECRET>`。
3. 非生产环境若未配置 `CRON_SECRET`，允许本地手动调用以便调试。

涉及路由：
- `/api/cron/cleanup-analytics`
- `/api/cron/flush-analytics-outbox`
- `/api/cron/cleanup-analytics-outbox`
- `/api/cron/cleanup-share-avatars`

## 5. 关键优化点（v3 已实施）

1. **严格化 CRON_SECRET**
- 解决“生产漏配时 cron 接口可被未授权访问”的风险。

2. **新增 outbox 清理**
- 防止 `analytics_outbox` 在导出成功后无限增长。

3. **错峰调度**
- 避免多个任务同一时刻打满 DB/外部 API。

4. **share 头像清理移除 5 天硬窗口**
- 旧逻辑仅清理近 5 天过期数据，若 cron 停摆较久会漏清。
- 新逻辑默认清理所有已过期记录（可选 lookback）。

## 6. 上线前检查清单

1. 环境变量
- `CRON_SECRET`
- `POSTHOG_API_KEY`
- `POSTHOG_HOST`
- `ANALYTICS_OUTBOX_BATCH_SIZE`
- `ANALYTICS_OUTBOX_RETENTION_DAYS`

2. Vercel Dashboard
- 确认 4 个 cron 都已注册
- 手动 `Run Now` 验证每个任务返回 `success: true`

3. 日志与数据验证
- `flush-analytics-outbox`：持续有 exported 结果，失败可重试
- `cleanup-analytics`：有删除计数（或 0 但执行成功）
- `cleanup-analytics-outbox`：有 `outboxDeleted` 计数
- `cleanup-share-avatars`：`checked/deleted/cleared` 数值合理

## 7. 运行建议（0->1 阶段）

1. 每周检查一次 `analytics_events` 和 `analytics_outbox` 行数增长。
2. 若出库延迟升高，优先调大 `ANALYTICS_OUTBOX_BATCH_SIZE`，再考虑提升 flush 频率。
3. 若 DB 压力明显增大，可先缩短 BUSINESS 留存窗口（如 90 -> 60 天），再评估更长期归档方案。

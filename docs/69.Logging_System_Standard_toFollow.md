# Logging 规范与统一标准

## 覆盖范围

1. Next.js 服务端：RSC / Route Handlers / Server Actions / DAL
2. Worker：任务执行、QStash handler、策略逻辑
3. 浏览器端 UI：交互、SSE 调试、用户态异常
4. LLM 与 RAG 调试输出
5. Sentry：错误与异常平台，不是常规运行日志平台

## 开关与启用方式

### Next.js

`.env.local`：

1. `LOG_INFO=true` 开启服务端 info 日志
2. `LOG_DEBUG=true` 开启服务端 debug 日志与 `logDebugData`
3. `LLM_DEBUG=true` 开启 LLM 文件调试输出
4. `NEXT_PUBLIC_LOG_DEBUG=true` 开启 UI 日志
5. `NEXT_PUBLIC_SSE_DEBUG=true` 开启 SSE 日志
6. `NEXT_PUBLIC_SENTRY_ENABLED=true` 显式启用浏览器侧 Sentry
7. `SENTRY_ENABLED=true` 显式启用服务端 Sentry

### Worker

`worker/.env` 或运行环境：

1. `LOG_INFO=true`
2. `LOG_DEBUG=true`
3. `LLM_DEBUG=true`
4. `SENTRY_ENABLED=true`

## 结构化日志标准

所有日志都应尽量携带结构化字段：

1. `reqId`
2. `route`
3. `phase`
4. `serviceId / userKey / taskId`
5. `durationMs / latencyMs`
6. `tokens / cost`
7. `error`

视场景补充：

1. `modelId / provider / tier / queue`
2. `status / code / isTrial`
3. `stage / traceId`
4. `runtime` (`web` / `worker`)

## 行为规范

1. 业务代码中禁止直接使用 `console.*`
2. 唯一允许的 console 出口是统一 logger 或测试代码
3. 服务端与 Worker 使用 `logInfo/logDebug/logError`
4. UI 使用 `uiLog` / `sseLog`
5. 文件调试输出统一走 `logDebugData`
6. 生产环境不得记录敏感内容与个人隐私字段
7. `error` 必须尽量传原始 `Error` 对象，避免丢失 stack

## Sentry 规范

### 事实边界

1. `logError` 会进入 Sentry
2. `logInfo/logDebug` 不会自动进入 Sentry
3. Sentry 用于异常、错误、显式 handled error
4. 普通运行日志仍以结构化控制台 / 文件调试为主

### 推荐做法

1. 常规流程日志：`logInfo`
2. 低层调试：`logDebug`
3. 抛错或失败：`logError`
4. 需要显式把非致命异常送进 Sentry：`captureHandledError`
5. 需要补用户、标签、上下文：`setUser / setTag / setContext`

### 禁止做法

1. 用 `captureMessage` 代替运行日志
2. 把 info/debug 级别日志批量灌进 Sentry
3. 在业务层混用 `console.error` 与 `logError`

## 常见示例

### 服务端

```ts
logInfo({
  reqId: serviceId,
  route: 'workbench/page',
  phase: 'service_fetch',
  serviceId,
  userKey: userId,
  durationMs: Date.now() - startedAt,
})
```

### 错误

```ts
try {
  await runTask()
} catch (error) {
  logError({
    reqId: taskId,
    route: 'worker/execute',
    phase: 'run_task_failed',
    taskId,
    serviceId,
    error,
  })
}
```

### handled error 进入 Sentry

```ts
await captureHandledError(new Error('resume_render_timeout'), {
  tag: ['runtime', 'web'],
  section: ['resume_render', { serviceId, phase: 'timeout' }],
  extra: { serviceId, taskId },
})
```

### UI

```ts
uiLog.error('resume_auto_save_failed', { error: errorMessage })
```

### LLM 调试文件

```ts
logDebugData('resume_summary_output', {
  output: JSON.stringify(result),
  meta: { serviceId, taskId },
  latencyMs,
})
```

## 与 Feedback 的关系

1. Founder feedback 只在 `bug + sentryEventId` 场景下展示 `Open Sentry`
2. `feature` / `confusion` 不应伪造 Sentry 链接
3. 创建服务态、空态、认知问题如果没有异常，不应出现 Sentry 深链
4. founder 排查时，先看 Sentry，再看 Replay，再看业务数据

## 迭代要求

1. 新增模块必须复用统一入口，不新增其他日志工具
2. 新增日志字段必须保持结构化与稳定命名
3. 对关键链路必须保证 `reqId / route / phase` 可追踪
4. 需要进入 Sentry 的信号必须明确说明原因，避免重新引入噪音

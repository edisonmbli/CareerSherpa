# Founder Inbox Slack Message Structure V2

## 1. 目标

Slack 消息的目标不是“把所有字段平铺给 founder 看”，而是按以下顺序组织：

1. 5 秒内看懂：这是什么反馈、发生在哪、要不要优先处理。
2. 10 秒内知道先查哪里：Open App、PostHog、Sentry、业务数据。
3. 需要时再下钻：完整标识、计费线索、模型线索、调试信号仍然都在。

V2 采用 `Incoming Webhook / Slack Bot + Block Kit`，不引入 Slack interactivity。

## 2. 字段来源与可信度

### 2.1 旧字段问题

V1 中有 3 个字段容易误导：

| 字段 | V1 来源 | 问题 | V2 处理 |
| --- | --- | --- | --- |
| `tier` | workbench UI 侧根据 `tierOverride` 或 `quotaBalance < currentTaskCost` 推断 | 这是当前页面的 UI 提示，不是某次实际执行的计费真值 | 改为 `taskTierHint`，只作为 hint 展示 |
| `queueType` | 实际上传的是 `tierOverride` | 它不是 QStash queue，也不是 LLM worker queue | 标记为 legacy，只下沉到 Debug 区 |
| `lastCost` | `getLedgerSummaryByService(...).totalDelta` 的绝对值 | 这是 service 维度净 coin 变化，不是“最后一次扣费金额” | 改名为 `serviceNetCoinDelta` |

### 2.2 V2 新字段

| 字段 | 来源 | 可信度 | 用途 |
| --- | --- | --- | --- |
| `taskId` | 当前页面可推导时由前端透传 | 中到高 | 优先做精确计费匹配 |
| `taskTemplateId` | 当前 tab/status 对应模板 ID | 中到高 | 回查 `llm_usage_logs` 与扣费 fallback |
| `taskTierHint` | 当前 tab 的 UI 层 tier 推断 | 中 | 辅助 founder 快速判断，不当真值 |
| `posthogDistinctId` | 浏览器端 `posthog.get_distinct_id()` | 高 | 进入 person/timeline 的主锚点 |
| `posthogSessionId` | 浏览器端 `posthog.get_session_id()` | 中到高 | 辅助确认会话级别的 replay |
| `posthogReplayUrl` | 浏览器端 `posthog.get_session_replay_url(...)` | 高 | 直接跳转当前 session replay |
| `billingMode` | `coin_transactions` 查询结果 | 高 | 判断此反馈对应任务是否实际扣费 |
| `billingStatus` | `coin_transactions.status` | 高 | 判断是 `SUCCESS / PENDING / FAILED / REFUNDED` |
| `debitDelta` | `coin_transactions.delta` 绝对值 | 高 | 确认实际扣费值 |
| `modelId` | `llm_usage_logs.model_id` | 高 | 看这次任务到底用了什么模型 |
| `provider` | `llm_usage_logs.provider` | 高 | 快速定位模型供应商 |
| `deliveryMode` | 由投递链路写入：`direct / qstash` | 高 | 看消息经由哪条通道送达 |
| `dispatchTarget` | 固定 `/api/feedback/dispatch` | 高 | 明确是 founder feedback 的专用 dispatch |

## 3. DB Enrichment 设计

### 3.1 为什么可行

这部分 enrich 放在服务端 delivery/dispatch 阶段，而不是用户主提交链路前端：

1. 反馈量远低于 LLM 主请求量，是低频路径。
2. `coin_transactions` 对 `taskId`、`serviceId` 都有索引。
3. `llm_usage_logs` 对 `serviceId`、`taskTemplateId` 都有索引。
4. feedback 在生产默认可走 QStash，因此 enrich 不会阻塞用户提交动作。

### 3.2 查询策略

计费查询顺序：

1. 若有 `taskId`，优先按 `userId + serviceId + taskId + SERVICE_DEBIT + createdAt <= occurredAt` 查最近一条。
2. 若没有精确命中，再退回 `userId + serviceId + templateId + SERVICE_DEBIT + createdAt <= occurredAt`。
3. 命中则视为 `billingMode = paid`，否则视为 `free`。

关于 `taskId` 有一个重要边界：

1. 如果用户停留在当前 tab 且任务刚执行过，前端通常能拿到真实 taskId。
2. 如果用户已经执行到更深步骤，再回退到历史 tab，当前 service 级别的 `executionSessionId` 不一定还能代表那个历史 tab 的真实 taskId。
3. 这种情况下，V2 的策略是 `宁可不传 taskId，也不传错 taskId`。

模型查询顺序：

1. 按 `userId + serviceId + taskTemplateId + createdAt <= occurredAt` 查最近一条 `llm_usage_logs`。
2. 命中后读取 `provider + modelId`。

### 3.3 阻塞评估

结论：合理且可接受。

原因：

1. enrich 不在核心业务写入路径。
2. 查询是单 service / 单 user / 单 task 维度的小范围索引命中。
3. 最坏情况下 enrich 失败，只影响 founder inbox 的上下文完整性，不影响主业务结果。

## 4. QStash Transport Note

当前 feedback 若环境中已启用 QStash，会走：

1. `POST /api/feedback`
2. 同步写入 `feedback_inbox`
3. `publishJSON({ feedbackId, deliveryMode })`
4. 投递到 `/api/feedback/dispatch`
5. dispatch 从库里读取记录，并完成 Slack 发送

### 4.1 为什么看起来“没有指定 queue”

因为这里使用的是 QStash 的 HTTP destination 模式。它的核心语义是“把消息投到某个 URL”，而不是必须显式声明一个业务队列名。

所以：

1. 这条 feedback 消息不会自动进入 LLM worker 的业务语义队列。
2. 它和 LLM 任务共享的是同一个 QStash 项目的基础设施、重试和配额，而不是共享同一个 handler。
3. 当前隔离边界来自 `dispatch URL`，而不是 `queueType` 字段。

### 4.2 当前 best practice

MVP 阶段建议这样做：

1. feedback 使用独立 dispatch endpoint：`/api/feedback/dispatch`
2. `feedback_inbox` 作为唯一事实源
3. retries 保守，例如当前 `3`
4. Slack 里明确展示 `deliveryMode` 与 `dispatchTarget`
5. 不把 `queueType` 这种旧字段继续当作真实传输语义

### 4.3 什么时候再升级隔离

当出现以下任一情况时，再考虑给 feedback 单独拆 QStash project/token/flow control：

1. founder inbox 量显著升高
2. LLM 主任务吞吐明显更高，且你担心共享配额/速率
3. 需要更精细的重试和死信策略

当前阶段，不必为 feedback 单独上“专门 queue 名”而过度设计。

## 5. Block Kit V2 结构

当前推荐结构是“两段式”：

1. 主消息：精简 triage
2. thread reply：详细上下文

主消息固定为以下层次：

1. `Header`
2. `User message`
3. `Quick triage`
4. `Action hints`
5. `Context footer`

thread reply 固定为以下层次：

1. `Identifiers`
2. `Billing / Model`
3. `Page / Flow`
4. `Debug / Signals`

### 5.1 Header

格式：

```text
{emoji} Founder Inbox · {surface}/{tab} · {status}
```

示例：

```text
🐞 Founder Inbox · workbench/interview · INTERVIEW_COMPLETED
```

### 5.2 User message

只放用户原话，高亮显示，不与 context 混排。

### 5.3 Quick triage

只放 founder 首屏决策需要的字段：

1. `Type`
2. `Priority`
3. `Surface / Tab`
4. `Service Status`
5. `Tab Status`
6. `Billing`
7. `Model`
8. `Reply Email`
9. `Triage Hint`

### 5.4 Action hints

当前只保留轻动作：

1. `Open App`
2. `PostHog Person`
3. `PostHog Replay`
4. `Open Sentry`（仅 `bug + sentryEventId`）
5. `Path`
6. `Submitted At`
7. `Locale`

### 5.5 Context footer

主消息底部只保留一句：

```text
Details are in thread · service {serviceId} · feedback {feedbackId}
```

注意：

1. 只有 bot thread 模式真正可用时，主消息才会出现这句提示。
2. 若 bot 模式失败并回退到 webhook 单消息模式，则不会显示该提示，详细上下文会直接并入主消息。

### 5.6 Thread: Identifiers

保留排查入口，但不打断首屏：

1. `serviceId`
2. `taskId`
3. `taskTemplateId`
4. `feedbackId`
5. `userId`
6. `posthogDistinctId`
7. `posthogSessionId`
8. `sentryEventId`
9. `sentryRuntime`

### 5.7 Thread: Billing / Model

这里是 V2 新增的业务判断层：

1. `taskTierHint`
2. `currentTabCostPreview`
3. `billingMode`
4. `billingStatus`
5. `billingLookupSource`
6. `debit`
7. `provider`
8. `modelId`
9. `modelLookupSource`
10. `llmUsageCreatedAt`

### 5.8 Thread: Page / Flow

保留页面与时间线：

1. `URL`
2. `Page Title`
3. `Occurred At`
4. `User Agent`

### 5.9 Thread: Debug / Signals

这里保留完整性，但不打断 triage：

1. `currentStatus`
2. `matchReady`
3. `interviewReady`
4. `connected`
5. `viewport`
6. `timezone`
7. `deliveryMode`
8. `dispatchTarget`
9. `posthog app`
10. `posthog project`
11. `sentry project`
12. `legacy tier`
13. `legacy queueType`
14. `serviceNetCoinDelta`

## 6. Deep Link 生成逻辑

### 6.1 Replay

Replay link 不由服务端猜测，而是由浏览器端 PostHog SDK 在用户提交反馈时直接生成：

```ts
posthog.get_session_replay_url({
  withTimestamp: true,
  timestampLookBack: 30,
})
```

这意味着：

1. link 对应的是用户提交反馈当下的那段 session replay
2. URL 结构由 PostHog SDK 自己保证
3. 服务端只负责透传和展示

### 6.2 Person

Person link 的优先级如下：

1. 若配置了 `POSTHOG_PERSON_URL_TEMPLATE`，按模板渲染
2. 否则若拿到了 `POSTHOG_APP_BASE_URL + POSTHOG_PROJECT_ID`，按默认格式生成
3. 若上述都没有，但 replay URL 可解析出 app/project，则从 replay URL 中反推 app/project，再生成 person link

默认 person path 约定：

```text
{appBaseUrl}/project/{projectId}/person/{distinctId}
```

## 6.3 Slack thread reply 的配置前提

如果要启用“主消息 + thread 详情”，仅靠 incoming webhook 不够。

需要新增：

```bash
FEEDBACK_SLACK_BOT_TOKEN=xoxb-...
FEEDBACK_SLACK_CHANNEL_ID=C0123456789
```

原因：

1. 主消息发出后，需要拿到 Slack 返回的 `ts`
2. 只有拿到 `ts`，才能继续发 thread reply
3. incoming webhook 本身不适合作为 thread 两段式发送的唯一通道

Slack bot 至少需要：

1. `chat:write`
2. 必要时补 `chat:write.public`
3. bot 已加入目标频道

未配置 bot token，或 bot 发送失败时，系统自动退回“单消息含完整详情”的 webhook 模式。

### 6.4 Sentry

Sentry link 的显示条件比 PostHog 更严格：

1. 反馈类型必须是 `bug`
2. payload 中必须存在 `sentryEventId`
3. 当前反馈不属于纯 feature / confusion

生成顺序：

1. 若服务端能用 `SENTRY_AUTH_TOKEN` 解析 event，则生成稳定的 `Open Sentry` event link
2. 若解析不到，则退回到按 `eventId` 搜索的链接
3. 默认跳 `career-shaper-web`
4. 若 payload 标记为 `sentryRuntime=worker`，则跳 `career-shaper-worker`

## 8. 字段迁移表

| V1 展示字段 | V2 去向 |
| --- | --- |
| `tier` | `taskTierHint`，并下沉到 Billing / Model |
| `queueType` | `Legacy Queue Type`，下沉到 Debug / Signals |
| `lastCost` | 改名为 `serviceNetCoinDelta`，下沉到 Debug / Signals |
| `serviceId / userId / distinctId / sentryEventId` 平铺在顶部 | 收敛到 `Identifiers` |
| 所有 context 一口气平铺 | 按 `Quick triage / Identifiers / Billing / Model / Debug` 分层 |

## 9. 运营建议

1. 首屏先看 `Type / Priority / Billing / Model / Status`。
2. 若是 `bug + paid`，优先处理。
3. 若是 `confusion + free`，优先考虑文案、状态提示和 FAQ。
4. 若是 `feature + paid`，先归类到 roadmap，再看是否有复现价值。
5. 若是 `billingStatus = PENDING/FAILED/REFUNDED`，这条消息的优先级判断要和纯 paid 不同，先看账务与异步状态。

## 1. 当前状态

项目的 Sentry 接入已经不是空白状态，但此前属于“半集成”：

1. `@sentry/nextjs` 与 `@sentry/node` 已安装。

4. 本地已提供标准验证入口：
   - 浏览器客户端

   - Edge Route Handler
   - Worker 调试路由
5. Founder Feedback 在 `bug + sentryEventId` 场景下会自动生成 `Open Sentry` 链接。

## 3. 环境变量标准

### 3.1 Web

本地 `.env.local` 与 Vercel Web 需要：

| 变量名 | 说明 | 必需 |
| --- | --- | --- |
| `NEXT_PUBLIC_SENTRY_DSN` | Web 项目的 DSN | ✅ |
| `SENTRY_AUTH_TOKEN` | Source Maps 上传与服务端事件链接解析 | ✅ |
| `SENTRY_ORG` | Sentry 组织 slug | ✅ |
| `SENTRY_PROJECT` | Web 项目 slug，建议 `career-shaper-web` | ✅ |
| `SENTRY_ENVIRONMENT` | 环境名，例如 `development` / `preview` / `production` | ✅ |
| `SENTRY_RELEASE` | 版本号，建议使用 git commit SHA | ✅ |
| `SENTRY_BASE_URL` | 可选，默认 `https://sentry.io` | ❌ |
| `SENTRY_ENABLED` | 服务端显式启用开关，`true/false` | ❌ |
| `NEXT_PUBLIC_SENTRY_ENABLED` | 浏览器端显式启用开关，`true/false` | ❌ |
| `SENTRY_DEBUG` | 本地排障时开启 SDK debug | ❌ |
| `SENTRY_TRACES_SAMPLE_RATE` | tracing 采样率，`0-1` | ❌ |
| `SENTRY_PROFILES_SAMPLE_RATE` | profiling 采样率，`0-1` | ❌ |

### 3.2 Worker

Worker / VPS 环境需要：

| 变量名 | 说明 | 必需 |
| --- | --- | --- |
| `SENTRY_DSN` | Worker 项目的 DSN | ✅ |
| `SENTRY_WORKER_PROJECT` | Worker 项目 slug，建议 `career-shaper-worker` | ✅ |
| `SENTRY_ORG` | Sentry 组织 slug | ✅ |
| `SENTRY_AUTH_TOKEN` | 可选，用于 feedback 事件链接解析与 CI/CD 共用 | ✅ |
| `SENTRY_ENVIRONMENT` | 环境名 | ✅ |
| `SENTRY_RELEASE` | 版本号，建议与 Web 使用同一 release | ✅ |
| `SENTRY_BASE_URL` | 可选，默认 `https://sentry.io` | ❌ |
| `SENTRY_ENABLED` | Worker 显式启用开关 | ❌ |
| `SENTRY_DEBUG` | 本地排障时开启 SDK debug | ❌ |
| `SENTRY_TRACES_SAMPLE_RATE` | tracing 采样率 | ❌ |
| `SENTRY_PROFILES_SAMPLE_RATE` | profiling 采样率 | ❌ |

本地开发推荐位置：

1. `worker/.env`
2. 或 `worker/.env.local`

生产环境推荐位置：

1. VPS / Docker / PM2 / systemd 的运行环境
2. 不要把生产 worker 凭证回写到仓库根目录 env 文件

### 3.3 建议值

建议的最小配置：

```bash
NEXT_PUBLIC_SENTRY_DSN=https://...
SENTRY_DSN=https://...
SENTRY_AUTH_TOKEN=sntrys_...
SENTRY_ORG=your-org
SENTRY_PROJECT=career-shaper-web
SENTRY_WORKER_PROJECT=career-shaper-worker
SENTRY_ENVIRONMENT=development
SENTRY_RELEASE=local-dev
NEXT_PUBLIC_SENTRY_ENABLED=true
SENTRY_ENABLED=true
```

说明：

1. Web 与 Worker 应使用不同 DSN、不同 project slug。
2. Web 与 Worker 应共享同一个 `SENTRY_RELEASE`，便于按版本回溯。
3. `NEXT_PUBLIC_SENTRY_ENABLED` 控制浏览器侧；`SENTRY_ENABLED` 控制服务端与 Worker。
4. 若只在生产启用，可不填显式开关，代码会在 `NODE_ENV=production` 下自动启用。
5. `.env` 文件中只写最终值，不要写 `$(git rev-parse --short HEAD)` 这类动态命令。

## 4. Sentry 后台操作指南

### 4.1 创建项目

在 Sentry 后台执行：

1. 新建项目 `career-shaper-web`
   - 平台选 `Next.js`
2. 新建项目 `career-shaper-worker`
   - 平台选 `Node.js`
3. 复制两个项目的 DSN
4. 记录两个项目的 slug

### 4.2 创建 Auth Token

在 Sentry 后台创建一个组织级 token，用于：

1. 构建期 Source Maps 上传
2. feedback 中的 Sentry event link 解析

建议 token 至少具备：

1. `Project: Read`
2. `Project: Write`
3. `Release: Admin`
4. `Org: Read`

### 4.3 告警与过滤

建议至少配置：

1. Web 项目：新 issue 告警
2. Worker 项目：新 issue 告警
3. 关闭明显无价值的浏览器扩展噪音过滤后再测试一遍本地事件
4. 若本地测试看不到事件，优先检查 inbound filters 与 spike protection

### 4.4 Source Maps 上传验证

生产构建后在 Sentry 检查：

1. Release 是否存在
2. Event JSON 中是否存在 `debug_meta`
3. 项目 `Debug Files` 中是否能找到对应 debug id

Sentry 官方说明：release 和 environment 都可由环境变量驱动，source context/debug file 需要在事件产生前上传成功才会映射到新事件。[Sentry Next.js options](https://docs.sentry.io/platforms/javascript/guides/nextjs/configuration/environments/), [Sentry source context verification](https://sentry.zendesk.com/hc/en-us/articles/22549090273947-How-do-I-verify-my-events-are-set-up-properly-for-Source-Context), [Sentry source map timing](https://sentry.zendesk.com/hc/en-us/articles/28525099421723-Source-maps-are-uploaded-correctly-but-some-events-aren-t-mapped)

## 5. 代码接入约束

### 5.1 初始化入口

当前标准入口如下：

1. 浏览器：`instrumentation-client.ts`
2. Node.js / Edge：`instrumentation.ts`
3. 全局 React 错误边界：`app/global-error.tsx`
4. Worker：`worker/src/index.ts`

不要再新增 `sentry.client.config.ts` 一类旧式入口。当前项目已切到 App Router 推荐形态。

### 5.2 logger 职责边界

当前职责已经固定：

1. `logError`
   - `console.error`
   - `captureException`
2. `logInfo/logDebug`
   - 只写结构化控制台日志
   - 不自动写入 Sentry
3. 如果确实需要把非异常信号送进 Sentry：
   - 用 [lib/sentry/universal.ts](/Users/edisonmbli/Projects/CareerShaper/lib/sentry/universal.ts) 显式调用
   - 不要再把普通运行日志伪装成 issue

这是必要的噪音控制。Sentry 的 `captureMessage()` 也会计入事件，并进入 issue 流，不适合承接常规 info/debug 日志。[Sentry event type explanation](https://sentry.zendesk.com/hc/en-us/articles/33714371838491-Can-I-control-the-event-type-between-default-and-error), [Sentry info events still count](https://sentry.zendesk.com/hc/en-us/articles/32587897494043-Can-I-create-informational-events-that-won-t-count-as-errors)

### 5.3 上下文字段

建议在异常上报前补齐：

1. `user`
2. `reqId`
3. `route`
4. `phase`
5. `serviceId`
6. `taskId`
7. `runtime` (`web` / `worker`)

Web 用户上下文当前由 [PostHogIdentifyBridge.tsx](/Users/edisonmbli/Projects/CareerShaper/components/analytics/PostHogIdentifyBridge.tsx) 同步到 Sentry。

## 7. Feedback 与 Sentry 的联动

### 7.1 适用场景

只有以下场景才会在 founder feedback 中显示 `Open Sentry`：

1. 反馈类型是 `bug`
2. 页面存在最近一次 `sentryEventId`
3. 该反馈不是纯 feature/confusion

不适用场景：

1. 纯 feature request
2. confusion 但没有异常
3. 创建服务态、空态、纯认知问题但没有 Sentry event

### 7.2 链接策略

当前实现优先做稳定的 `event link`：

1. 若服务端能用 `SENTRY_AUTH_TOKEN` 解析到 group / event 信息，Slack 中显示 `Open event`
2. 若解析不到，则退回 `Find event` 搜索链接
3. 默认 bug feedback 先链接 Web 项目；若 feedback payload 带 `sentryRuntime=worker`，则链接 Worker 项目

关键文件：

1. [lib/sentry/links.ts](/Users/edisonmbli/Projects/CareerShaper/lib/sentry/links.ts)
2. [lib/feedback/delivery.ts](/Users/edisonmbli/Projects/CareerShaper/lib/feedback/delivery.ts)
3. [components/feedback/FounderFeedbackDialog.tsx](/Users/edisonmbli/Projects/CareerShaper/components/feedback/FounderFeedbackDialog.tsx)

### 7.3 founder 排查顺序

推荐固定顺序：

1. 先看 Slack 主消息里的 `Open Sentry`
2. 再看 `Open Replay`
3. 再回到业务页与数据库状态

原因：Sentry 适合先确认“是不是异常”，Replay 适合后确认“用户是怎么走到这一步的”。

# CareerShaper PostHog Configuration Playbook (v3.1)

本文档是 `#73` 的操作版补充，目标是让你上线后可以直接照着配置：
1. PostHog 项目配置
2. 10 个漏斗/看板配置
3. 3 个预警配置
4. 长线运营执行节奏

## 0. Scope

适用范围：
1. 当前 CareerShaper v3.1（DB truth + outbox -> PostHog）
2. 未来其它 0-1 项目（可复用本模板）

当前数据约束（必须先知道）：
1. `RESUME_SHARE_VIEW/RESUME_SHARE_CTA_CLICK` 与 `SERVICE_CREATED` 的 `distinct_id` 口径不同。
2. `LANDING_ATTRIBUTED` 与 `SERVICE_CREATED` 的 `distinct_id` 口径不同。
3. 因此跨这两类事件的“转化”优先使用 SQL Insight（按 `shareId/acqShareId` 或 `utm/acq*` join），不要强依赖普通 Funnel。

## 3.9 Funnel-9 Dashboard A（Resume Share Growth，SQL 真正转化）
类型：SQL Insight  
用途：按 `shareId` 计算 view/cta/service 真正转化  
SQL：

```sql
WITH e AS (
  SELECT
    event,
    toString(properties.shareId) AS share_id_view,
    toString(properties.acqShareId) AS share_id_service
  FROM events
  WHERE timestamp >= now() - INTERVAL 14 DAY
    AND event IN ('RESUME_SHARE_VIEW', 'RESUME_SHARE_CTA_CLICK', 'SERVICE_CREATED')
),
agg AS (
  SELECT
    coalesce(nullIf(share_id_view, ''), nullIf(share_id_service, '')) AS share_id,
    countIf(event = 'RESUME_SHARE_VIEW') AS views,
    countIf(event = 'RESUME_SHARE_CTA_CLICK') AS cta_clicks,
    countIf(event = 'SERVICE_CREATED') AS services
  FROM e
  GROUP BY share_id
)
SELECT
  share_id,
  views,
  cta_clicks,
  services,
  round(cta_clicks / nullIf(views, 0), 4) AS view_to_cta_rate,
  round(services / nullIf(cta_clicks, 0), 4) AS cta_to_service_rate
FROM agg
WHERE share_id IS NOT NULL
ORDER BY services DESC, views DESC
LIMIT 50;
```

## 3.10 Funnel-10 Dashboard B（Landing Attribution，SQL 真正归因）
类型：SQL Insight  
用途：按 `utm/channel` 计算 landing -> service 转化  
SQL：

```sql
WITH landings AS (
  SELECT
    toString(properties.channel) AS channel,
    toString(properties.utm_source) AS utm_source,
    toString(properties.utm_medium) AS utm_medium,
    toString(properties.utm_campaign) AS utm_campaign,
    count(*) AS landing_sessions
  FROM events
  WHERE timestamp >= now() - INTERVAL 14 DAY
    AND event = 'LANDING_ATTRIBUTED'
    AND toString(properties.touch) = 'first'
  GROUP BY channel, utm_source, utm_medium, utm_campaign
),
creates AS (
  SELECT
    toString(properties.acqChannel) AS channel,
    toString(properties.acqSource) AS utm_source,
    toString(properties.acqMedium) AS utm_medium,
    toString(properties.acqCampaign) AS utm_campaign,
    count(*) AS service_created
  FROM events
  WHERE timestamp >= now() - INTERVAL 14 DAY
    AND event = 'SERVICE_CREATED'
  GROUP BY channel, utm_source, utm_medium, utm_campaign
)
SELECT
  l.channel,
  l.utm_source,
  l.utm_medium,
  l.utm_campaign,
  l.landing_sessions,
  coalesce(c.service_created, 0) AS service_created,
  round(coalesce(c.service_created, 0) / nullIf(l.landing_sessions, 0), 4) AS landing_to_service_rate
FROM landings l
LEFT JOIN creates c
  ON l.channel = c.channel
 AND l.utm_source = c.utm_source
 AND l.utm_medium = c.utm_medium
 AND l.utm_campaign = c.utm_campaign
ORDER BY service_created DESC, landing_sessions DESC
LIMIT 100;
```

## 8. PostHog UI 点选路径（截图导向）

1. 左侧 `Product analytics` -> `Dashboards`
2. 点击 `New dashboard`
3. 输入名称：
- `Resume Share Growth v3.1`
- `Landing Attribution v3.1`
4. 选择可见性：团队可见（推荐）

### 8.2 创建 Funnel Insight（通用）
1. 左侧 `Product analytics` -> `Insights`
2. 点击 `New insight` -> 选择 `Funnels`
3. 在 Step 1/2/... 选择事件名
4. 在每个 Step 下设置 Filters（如 `type=detailed`）
5. `Conversion window` 设为本文建议值（24h / 7d）
6. `Breakdown` 选择属性（如 `acqChannel`）
7. 右上角 `Save` -> 选择目标 dashboard

### 8.3 创建 Trends Insight（通用）
1. `Insights` -> `New insight` -> `Trends`
2. 添加 1~2 个事件
3. 设置时间粒度（日/小时）
4. 需要拆分时设置 `Breakdown`
5. `Save` 到目标 dashboard

### 8.4 创建 SQL Insight（通用）
1. `Insights` -> `New insight` -> `SQL`
2. 粘贴本手册 SQL
3. 先点 `Run`
4. 确认结果列名/排序正确
5. `Save` 到目标 dashboard

### 8.5 创建 Alert（预警）
1. 打开要监控的 Insight
2. 右上角 `Create alert`
3. 配置阈值和观察窗口
4. 配置通知渠道（Slack / Email）
5. 保存并手动触发一次测试通知

## 11. 常见配置问题与排查

### 11.1 看板没数据
1. 先查 Events 是否有原始事件
2. 查时间范围是否过窄（先用 Last 30 days）
3. 查事件属性拼写是否一致（`acqChannel` vs `acq_channel`）

### 11.2 Funnel 转化明显不合理
1. 检查是否跨了不同 `distinct_id` 口径
2. 跨主体漏斗改用 SQL（本手册 3.9 / 3.10）

### 11.3 SQL 报字段不存在
1. 在事件详情确认属性是否已进 PostHog（properties）
2. 等待 outbox flush 一轮后重试
3. 用 `toString(properties.xxx)` 防止类型不一致

### 11.4 Alert 频繁误报
1. 拉长观察窗口（15m -> 60m）
2. 改成连续多窗口触发
3. 用分渠道阈值替代全局阈值

### 11.5 团队理解口径不一致
1. 每个卡片标题加后缀：`(Funnel)` / `(SQL Truth)` / `(Trend Only)`
2. 在卡片描述写清口径与限制

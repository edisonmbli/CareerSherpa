# Sentry Alert Config

**先说结论**
生产告警我建议分成两层：

1. **现在就做，低复杂度高收益**
2. **等生产跑一段时间后再加，避免噪音**

因为 Sentry 官方当前最实用的基础能力还是两类：

- `Issue Alerts`
  - 例如新 issue、回归、某 issue 在一段时间内出现很多次
- `Metric Alerts`
  - 例如错误数、失败率这类聚合指标

官方文档里，Issue Alert 明确支持：
- `A new issue is created`
- `The issue changes state from resolved to unresolved`
- `The issue is seen more than value times in interval`  
来源：[Create an Issue Alert Rule for a Project](https://docs.sentry.io/api/alerts/create-an-issue-alert-rule-for-a-project/)

---

## 我建议你现在立刻配置的规则

### A. Web 新 Issue
**目的**
- 新的前端/Next.js 错误第一时间知道

**规则**
- Project: `careersherpa-web`
- Environment: `production`
- Condition: `A new issue is created`
- Action: 发到你的 Slack / Email
- Frequency: `30` 或 `60` 分钟

**理由**
- 这是最值钱、最不容易出噪音的一条

---

### B. Worker 新 Issue
**目的**
- worker 新异常第一时间知道

**规则**
- Project: `careersherpa-worker`
- Environment: `production`
- Condition: `A new issue is created`
- Action: Slack / Email
- Frequency: `30` 或 `60` 分钟

**理由**
- worker 的错误通常比 web 更隐蔽
- 这条规则的性价比很高

---

### C. Web 回归告警
**目的**
- 已解决的问题重新出现时知道

**规则**
- Project: `careersherpa-web`
- Environment: `production`
- Condition: `Regression`
- Action: Slack / Email

**理由**
- 这类问题通常非常值得关注
- 因为它意味着修复失效、或某次 release 引入回滚

官方通知里也单独把 regression 作为 workflow 通知类型列出：[Notifications](https://docs.sentry.io/hosted/learn/notifications/)

---

### D. Worker 回归告警
**规则**
- Project: `careersherpa-worker`
- Environment: `production`
- Condition: `Regression`

**理由**
- 对 worker 更有价值，因为后台任务的回归用户往往不会立刻显式报错

---

### E. 高频重复触发告警
这个就是你提到的“高频回归 / 某类问题突然变多”。

**推荐做法**
先用 Issue Alert，不要一上来就上复杂 Metric Alert。

**规则示例**
- Project: `careersherpa-web`
- Condition: `Issue is seen more than 20 times in 1h`
- Environment: `production`

Worker 也做一条：
- Project: `careersherpa-worker`
- Condition: `Issue is seen more than 10 times in 1h`

**理由**
- 这是官方原生支持的，简单直接
- 能很好覆盖“某个错误突然爆发”

---

## 我建议先不要立刻做的规则

### 1. “某个 release 后激增” 的自动告警
这个想法是对的，但**现在不建议优先做成自动规则**。

原因：
- `release` 维度对你现在更适合做**发布后人工验证**
- 如果把告警直接绑到 release spike，通常要更精细的 metric/filter 设计
- 现阶段复杂度高于收益

**更合适的做法**
每次生产发布后，固定做一个 `release verification checklist`：

1. 去 `careersherpa-web`
2. 过滤：
   - `environment:production`
   - `release:<这次版本>`
   - 时间 `since release`
3. 看：
   - 新 issue 数
   - regression 数
   - stack/source map 是否正常
4. 对 `careersherpa-worker` 做同样检查

也就是说，**release 先做验证，不急着做复杂自动告警**。

---

### 2. “Worker 某类任务连续失败” 的精细告警
这个方向有价值，但前提是你的事件里必须有稳定 tag。

比如你要能稳定按这些维度筛：
- `route`
- `phase`
- `taskTemplateId`
- `strategy`

如果这些标签现在还没完全稳定，就先别做特别细的 worker-specific alerts，不然你会很快收到一堆不够精准的通知。

**建议顺序**
- 先跑一段时间
- 看 worker 项目里最常出现的是按什么 tag 聚类
- 再决定要不要加：
  - `summary task failures`
  - `match task failures`
  - `customize task failures`

---

## 现阶段最优的告警组合

我建议你先只配 **6 条**：

1. `careersherpa-web` 新 issue
2. `careersherpa-worker` 新 issue
3. `careersherpa-web` regression
4. `careersherpa-worker` regression
5. `careersherpa-web` 高频 issue（例如 `>20 in 1h`）
6. `careersherpa-worker` 高频 issue（例如 `>10 in 1h`）

这套规则：
- 覆盖新问题
- 覆盖回归
- 覆盖爆发性问题
- 复杂度低
- 几乎不会过度设计

## Release 验证怎么做

这个是另外一块，但同样值得现在投入。

### 每次生产发布后固定检查
**Web**
- Project: `careersherpa-web`
- Filter:
  - `environment:production`
  - `release:<当前 release>`

检查：
1. 是否有事件进入
2. stack trace 是否有源码映射
3. 是否出现新 issue
4. 是否出现 regression

**Worker**
- Project: `careersherpa-worker`
- Filter:
  - `environment:production`
  - `release:<当前 release>`

检查同上。

### 核心目的
不是“看很多图”，而是回答 4 个问题：

1. 这次 release 是否真的被 Sentry 识别到了
2. Web / Worker 是否用了同一个 release
3. source maps 是否工作正常
4. 这次发布是否引入了新问题/回归

---

## 我的建议优先级

**现在就做**
1. 新 issue 告警
2. regression 告警
3. 高频 issue 告警
4. 发布后 release verification checklist

**暂缓**
1. release spike 自动告警
2. Worker 某类任务连续失败的精细规则
3. ownership 自动分配

---

## 一句话总结
你要的方向是对的，但现阶段最有性价比的不是“把规则做很细”，而是：

- 先把 `新 issue + regression + 高频 issue`
- 配到 `web / worker` 两个项目
- 再把 `release verification` 固化成每次发布后的固定动作

这样投入小，但收益已经很高。

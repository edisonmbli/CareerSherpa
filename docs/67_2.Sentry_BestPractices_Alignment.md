# 1 **Findings**

1. `P1` 统一错误包装层正在把大量预期的 `4xx/校验/权限` 失败写进 Sentry，和 67/69 的边界定义冲突。  
   关键点在 [lib/api/utils.ts:54](/Users/edisonmbli/Projects/CareerShaper/lib/api/utils.ts:54), [lib/actions/utils.ts:51](/Users/edisonmbli/Projects/CareerShaper/lib/actions/utils.ts:51), [lib/api/auth-wrapper.ts:71](/Users/edisonmbli/Projects/CareerShaper/lib/api/auth-wrapper.ts:71)。这些入口会把 `missing_fields`、`invalid_request`、`unauthorized`、`forbidden`、`not_found`、`rate_limited` 等统一走 `logError -> captureException`。按 [docs/67.Logging_Migrate_to_Sentry_Guide.md:188](/Users/edisonmbli/Projects/CareerShaper/docs/67.Logging_Migrate_to_Sentry_Guide.md:188) 和 [docs/69.Logging_System_Standard_toFollow.md:103](/Users/edisonmbli/Projects/CareerShaper/docs/69.Logging_System_Standard_toFollow.md:103)，Sentry 应该承接异常与显式 handled error，而不是常规客户端失败。这个问题是系统性的，会上线后直接放大 issue 噪音。

2. `P1` 认证/安全链路里多处“预期拒绝”也被当成异常送进 Sentry。  
   典型位置是 [lib/security/middleware.ts:147](/Users/edisonmbli/Projects/CareerShaper/lib/security/middleware.ts:147), [lib/security/middleware.ts:195](/Users/edisonmbli/Projects/CareerShaper/lib/security/middleware.ts:195), [lib/security/middleware.ts:225](/Users/edisonmbli/Projects/CareerShaper/lib/security/middleware.ts:225), [lib/actions/auth.ts:42](/Users/edisonmbli/Projects/CareerShaper/lib/actions/auth.ts:42), [lib/actions/auth.ts:115](/Users/edisonmbli/Projects/CareerShaper/lib/actions/auth.ts:115), [lib/actions/utils.ts:210](/Users/edisonmbli/Projects/CareerShaper/lib/actions/utils.ts:210)。`suspicious_user_key`、`invalid_headers`、未登录、用户尚未同步到本地表、越权访问，这些更像审计/结构化日志场景，不应该默认成为 Sentry issue。否则真实生产异常会被攻击流量、匿名请求和同步延迟淹没。

3. `P2` 多个 capture 点没有保留原始 `Error`，导致 Sentry 里 stack 和现场会显著变差。  
   代表位置是 [lib/api/utils.ts:72](/Users/edisonmbli/Projects/CareerShaper/lib/api/utils.ts:72), [lib/actions/utils.ts:76](/Users/edisonmbli/Projects/CareerShaper/lib/actions/utils.ts:76), [lib/api/auth-wrapper.ts:74](/Users/edisonmbli/Projects/CareerShaper/lib/api/auth-wrapper.ts:74), [lib/actions/auth.ts:67](/Users/edisonmbli/Projects/CareerShaper/lib/actions/auth.ts:67)。这些地方传给 `logError` 的主要是错误码或错误消息字符串，而不是原始异常对象。按 [docs/69.Logging_System_Standard_toFollow.md:97](/Users/edisonmbli/Projects/CareerShaper/docs/69.Logging_System_Standard_toFollow.md:97)，`error` 应尽量传原始 `Error`。现在一旦真有内部错误从这些包装层出去，Sentry 看到的会是弱化后的“包装错误”，不利于定位根因。

4. `P2` `app/global-error.tsx` 直接调用 `Sentry.captureException`，没有补齐当前规范要求的标准上下文。  
   位置在 [app/global-error.tsx:12](/Users/edisonmbli/Projects/CareerShaper/app/global-error.tsx:12)。这里至少缺了 `runtime=web`、`route/global_error_boundary`、`phase`、`digest` 这类当前文档要求的结构化信息，也绕过了现有 helper。结果是全局 React 边界错误虽然能进 Sentry，但检索维度和现场质量比 dev debug route 弱很多。对照来看，[app/api/dev/sentry/server/route.ts:8](/Users/edisonmbli/Projects/CareerShaper/app/api/dev/sentry/server/route.ts:8), [app/api/dev/sentry/edge/route.ts:8](/Users/edisonmbli/Projects/CareerShaper/app/api/dev/sentry/edge/route.ts:8), [worker/src/index.ts:125](/Users/edisonmbli/Projects/CareerShaper/worker/src/index.ts:125) 已经是更符合规范的正例。

5. `P3` 当前没有发现 `captureMessage` 被直接滥用，这点是好的，但 `logError` 的默认语义仍然过宽。  
   搜索结果显示直接 Sentry 调用基本集中在 helper、debug route、global error、feedback `lastEventId` 读取这几类入口；没有看到把 info/debug 批量写入 Sentry 的新代码。这说明 67/69 的第一轮收口方向是对的，问题主要剩在“哪些失败应不应该走 `logError`”和“走了 `logError` 后现场是否够强”。

**结论**
- 当前最大的质量风险不是“到处乱用 Sentry API”，而是**通用 wrapper 和 auth/security 层把预期失败误归类成异常**。
- 当前最大的可观测性风险不是“完全没上下文”，而是**不少真正进入 Sentry 的事件只有字符串化错误，缺 stack，且没有统一 tags/context**。
- 所以提优的优先级应先做“降噪”，再做“补现场”。


# 2 **Implemantation Plan**

# Sentry 使用质量提优计划

## Summary
- 把 Sentry 的责任边界收紧到“真正异常、非预期失败、显式 handled error”，把常规 `4xx/权限/校验/同步延迟` 退回结构化日志或审计日志。
- 统一提高真正异常的现场质量：保留原始 `Error`，补齐 `reqId / route / phase / runtime / userKey / serviceId / taskId`。
- 重点先改系统性入口，不从零散业务点下手。

## Key Changes
- 重构通用包装层：
  - `lib/api/utils.ts`
  - `lib/actions/utils.ts`
  - `lib/api/auth-wrapper.ts`
  规则固定为：`4xx/expected business failure -> logInfo 或新增 logWarn（不进 Sentry）`，`5xx/internal/unhandled -> logError（进 Sentry）`。
- 收紧 auth/security 链路：
  - `lib/security/middleware.ts`
  - `lib/actions/auth.ts`
  - `validateUserAccess`
  将未登录、无权限、header 非法、同步延迟、可预期拒绝改成结构化审计/普通日志；只有中间件自身崩溃、校验器抛未预期异常时才进 Sentry。
- 保留原始异常对象：
  - 所有 `handleApiError/handleActionError` 风格入口都应优先把原始 `Error` 透传到 `logError`
  - 错误码、HTTP status、业务 code 放到附加字段，不替代 `error`
- 补标准上下文：
  - `app/global-error.tsx` 改为走 Sentry helper，附带 `runtime=web`、`route=app/global-error`、`phase=render_boundary`、`digest`
  - 对真正要进 Sentry 的入口，至少保证 `reqId / route / phase` 稳定，能拿到时继续补 `userKey / serviceId / taskId`
- 明确 Sentry 联动场景不扩张：
  - Founder feedback 继续只在 `bug + sentryEventId` 时展示 `Open Sentry`
  - 不把 feature/confusion 或创建态反馈强行接 Sentry

## Test Plan
- 预期不再进入 Sentry：
  - `missing_fields`
  - `invalid_request`
  - `unauthorized`
  - `forbidden`
  - `not_found`
  - `rate_limited`
  - 新用户同步延迟
  - 资源越权访问
- 预期仍进入 Sentry：
  - 包装层未识别的内部异常
  - 中间件自身异常
  - React global error
  - 显式 `captureHandledError` 的调试/业务场景
- 现场质量验证：
  - Sentry 中真实错误应保留原始 stack
  - event tags/context 至少包含 `route / phase / runtime`
  - 能拿到业务上下文的事件包含 `reqId / userKey / serviceId / taskId`
- 回归验证：
  - `/zh/sentry`
  - `/api/dev/sentry/server`
  - `/api/dev/sentry/edge`
  - worker `/debug/sentry`
  - founder feedback bug 链接保持可用

## Assumptions
- 默认采用“预期 4xx 不进 Sentry”的强约束，即使它们在业务上重要，也先留在结构化日志/审计日志，不占用 issue 流。
- 若代码库当前没有 `logWarn`，可先用 `logInfo` 承接这些 expected failures；是否新增 `logWarn` 作为第二步收敛。
- 本轮优先改系统性入口，不追求一次性清理所有业务点，只要把 wrapper/auth/security 三个主噪音源收住，Sentry 质量就会明显提升。

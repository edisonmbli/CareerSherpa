

## Sentry 后台怎么高效用

你现在看到信息量很大，这是正常的。  
高效使用不是“全看”，而是固定一套筛选和排查顺序。

我建议你只用这套流程。

### 1. 先只看两个项目
- `careersherpa-web`
- `careersherpa-worker`

不要从 “All Projects” 开始看。

### 2. 先只看 `Issues`
不要先看 `Insights`。

`Insights` 更偏：
- 趋势
- 性能
- 会话健康

而 founder / 独立开发者日常排查，优先应该是：
- `Issues`

### 3. 先固定过滤条件
每次先加这几个过滤维度：

- `environment:development` 或 `environment:production`
- `release:<当前版本>`
- 最近时间范围，例如 `24h` 或 `7d`

这样能先把噪音砍掉一大截。

### 4. 排查顺序固定
我的建议顺序：

1. `Slack founder inbox`
2. `Open Sentry`
3. 看 issue / event
4. 再去 `Open Replay`
5. 最后再看业务数据 / DB / worker logs

也就是：

- **先确认是不是异常**
- **再看用户怎么走到这一步**
- **最后看系统内部数据**

这比直接在 PostHog / DB 里乱翻效率高很多。

### 5. 在 Sentry 里先看什么字段
打开单条 event 后，优先只看：

1. `Error message`
2. `Stack trace`
3. `Tags`
   - `runtime`
   - `route`
   - `phase`
   - `environment`
   - `release`
4. `Contexts / Extra`
   - `reqId`
   - `serviceId`
   - `taskId`
   - `userKey`

不要一开始就把每个 panel 全展开。

### 6. 你最常用的不是 Event 列表，而是 Issue 聚类
对你这种产品，真正高效的是：

- 看这个问题是不是已经发生过
- 是否集中在同一个 release
- 是否只影响某个 route / 某个 provider / 某个 worker task

所以大多数时候：
- 先看 `Issue`
- 有需要再点进具体 `Event`

### 7. 如何避免被信息爆炸拖住
你可以固定只回答这 4 个问题：

1. 是 Web 还是 Worker？
2. 是哪条 route / 哪个 phase？
3. 是不是当前 release 新引入的？
4. 需要先看 Replay 还是先看服务端日志？

只要这 4 个答案明确了，排查路径通常就出来了。

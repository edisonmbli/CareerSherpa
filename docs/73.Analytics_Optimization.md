# CareerShaper Analytics Optimization (v2 -> v3)

## 0. 目标与定位

Analytics 在本项目中的定位是三层分工：

1. Vercel Analytics：PV/UV 与页面级流量。
2. DB 业务表：最终结果（Service/Match/Customize/Interview 的落库状态）。
3. Internal Analytics（本方案）：过程态观测 + 瓶颈诊断 + 增长漏斗。

本次文档同时覆盖：
- 已落地的 v2（DB 真理源 + Outbox -> PostHog 同步）
- 新增的 v3（前端 PostHog Product Layer：Autocapture/Replay/Survey + identify 聚合）

操作配置手册请见：`docs/73_1.Analytics_PostHog_Config.md`

---

## 1. v2 已落地基线（当前事实）

### 1.1 数据面
1. `analytics_events` 作为主写真理源（在线诊断 + 业务 join）。
2. `analytics_outbox` 同事务写入，用于异步外发到 PostHog。
3. Cron 定时 flush：`/api/cron/flush-analytics-outbox`。

### 1.2 事件治理
1. 统一入口：`trackEvent` / `logEvent`。
2. `TASK_ENQUEUED` 仅 Producer 上报（Action 重复上报已移除）。
3. Guard/Worker 生命周期事件已标准化：
- `WORKER_JOB_STARTED`
- `WORKER_JOB_FINISHED`
- `WORKER_GUARDS_BLOCKED`
- `WORKER_PROVIDER_NOT_CONFIGURED`

### 1.3 隐私最小化
1. Waitlist 邮箱仅存 hash（analytics 不存明文）。
2. Share referrer 仅保留域名（不落完整 URL/query）。

## 2. v3 目标架构（PostHog Product Layer）

## 2.1 文字架构图

```text
[Browser]
  └─ posthog-js (Autocapture + Replay + Surveys)
       └─ identify(userId)

[Next.js Actions / Worker]
  └─ trackEvent/logEvent
       └─ analytics_events (DB truth)
       └─ analytics_outbox (same transaction)
            └─ cron flush -> PostHog Capture API

[Vercel Analytics]
  └─ PV/UV only
```

## 2.2 关键原则
1. DB 继续是真理源，不反向依赖 PostHog 作为业务判定。
2. PostHog 用于产品分析层：漏斗、回放、调研、实验。
3. 前端 SDK 只在 `Workbench + Profile` 范围启用，避免全站采集噪音和隐私风险。

## 3. v2 -> v3 差异表

| 维度 | v2 | v3 |
|---|---|---|
| 前端行为采集 | 无 | `Autocapture + Session Replay + Surveys` |
| 用户聚合 | 仅后端事件 userId | 增加 `identify(userId)` 聚合前后端事件 |
| 录屏能力 | 无 | 有（仅 Workbench + Profile） |
| 漏斗入口事件 | 缺 `CUSTOMIZE_SESSION_STARTED` / `CUSTOMIZE_SHARE_CLICKED` | 补齐两事件 |
| 隐私策略 | email/referrer 去敏 | 再加 replay 采集边界 + mask/block 规则 |

## 4. 事件体系 v3（增量）

## 4.1 新增业务事件
1. `CUSTOMIZE_SESSION_STARTED`
- 触发点：`customizeResumeAction` 入队成功后。
- 用途：Step2 定制漏斗入口。

2. `CUSTOMIZE_SHARE_CLICKED`
- 触发点：`generateShareLinkAction` 成功后。
- 语义：生成分享链接成功（稳定入口）。
- 用途：裂变漏斗起点。

## 4.2 命名策略
不新增 `DETAILED_*` 平行事件名，继续使用统一事件 + 属性过滤：
- 上传：`RESUME_UPLOAD_ACCEPTED` + `type='detailed'`
- 解析：`RESUME_PARSE_COMPLETED` + `templateId='detailed_resume_summary'` + `success=true`

## 5. Seven Funnel Definitions（PostHog 标准定义）

> 说明：事件名以系统真实埋点为准；所有漏斗都可直接在 PostHog Funnel UI 配置。

### 5.1 通用简历上传漏斗
`RESUME_UPLOAD_ACCEPTED(type='resume') -> RESUME_PARSE_COMPLETED(templateId='resume_summary', success=true)`
- 业务用途：衡量基础资产构建成功率。
- 关键观察：解析成功率、解析耗时分布。
- 常见断点：上传文本质量低、解析失败码集中。

### 5.2 详细履历上传漏斗
`RESUME_UPLOAD_ACCEPTED(type='detailed') -> RESUME_PARSE_COMPLETED(templateId='detailed_resume_summary', success=true)`
- 业务用途：衡量高级资产构建率。
- 关键观察：详细履历渗透率、解析成功率。
- 常见断点：用户不愿补充详细履历、长文本质量问题。

### 5.3 核心业务深度漏斗
`SERVICE_CREATED -> MATCH_GENERATED -> CUSTOMIZE_GENERATED(success=true) -> INTERVIEW_COMPLETED`
- 业务用途：衡量单个 service 的完整使用深度。
- 关键观察：各阶段转化、阶段耗时、失败分布。
- 常见断点：匹配完成后不进入 customize/interview。

### 5.4 Step1 匹配漏斗
`SERVICE_CREATED -> MATCH_GENERATED`
- 业务用途：衡量核心第一跳可达率。
- 关键观察：匹配成功率、失败码、backpressure 影响。
- 常见断点：队列拥塞、模型配置缺失、解析前置失败。

### 5.5 Step2 定制漏斗
`CUSTOMIZE_SESSION_STARTED -> CUSTOMIZE_GENERATED(success=true) -> CUSTOMIZE_EXPORT_CLICKED`
- 业务用途：衡量“启动定制 -> 产出 -> 导出”的真实价值链路。
- 关键观察：生成成功率、导出率。
- 常见断点：生成失败或用户只看不导出。

### 5.6 Step3 面试漏斗
`INTERVIEW_SESSION_STARTED -> INTERVIEW_COMPLETED`
- 业务用途：衡量面试模块完成率。
- 关键观察：完成率、失败率、失败码。
- 常见断点：上下文不完整、模型超时。

### 5.7 裂变召回漏斗
`CUSTOMIZE_SHARE_CLICKED -> RESUME_SHARE_VIEW -> RESUME_SHARE_CTA_CLICK`
- 业务用途：衡量分享传播效率与召回价值。
- 关键观察：分享点击->被查看->CTA 点击的转化。
- 常见断点：分享入口低、落地页吸引力不足。

### 5.8 裂变召回漏斗
`CUSTOMIZE_SHARE_CLICKED -> RESUME_SHARE_VIEW -> RESUME_SHARE_CTA_CLICK -> USER_SIGNUP_COMPLETED -> SERVICE_CREATED`
- 业务用途：衡量分享传播效率与召回价值。
- 关键观察：分享点击->被查看->CTA 点击的转化。
- 常见断点：分享入口低、落地页吸引力不足。

### 5.10 Landing 转化漏斗
`LANDING_ATTRIBUTED -> SERVICE_CREATED`

- 业务用途：衡量 Landing 首触/末触归因。
- 关键观察：Landing 首触/末触 -> Service 创建。
- 常见断点：Landing 首触/末触低、Service 创建低。

## 6. Frontend SDK 接入约束（v3）

## 6.1 启用范围
1. 启用：`/workbench/*`、`/profile`
2. 不启用：营销页、公共分享页（避免额外噪音和隐私风险）

## 6.2 identify 聚合
在登录态调用：
- `identify(userId, { email?, plan?, locale?, analytics_scope })`

在登出或用户切换时调用：
- `reset()`

## 6.3 录屏隐私策略
1. `maskAllInputs=true`（输入框默认掩码）
2. `maskTextSelector` 包含：
- `[data-ph-mask]`
- `.ph-mask-text`
- `.resume-paper`
- `.public-resume-content`
3. `blockSelector` 包含：
- `[data-ph-block]`
- `.ph-no-capture`

## 7. PostHog 操作手册（从 0 到 1）

## 7.1 上线前（配置与验收）

### A. 开通与项目设置（US）
1. 注册 PostHog Cloud，创建项目（Region 选 US）。
2. 在 Project Settings 获取：
- Project API Key（前端）
- Personal API Key（如需管理 API）

### B. 环境变量配置

服务端（已有 v2）
- `POSTHOG_API_KEY`
- `POSTHOG_HOST=https://us.i.posthog.com`
- `ANALYTICS_OUTBOX_BATCH_SIZE`
- `CRON_SECRET`

前端（v3）
- `NEXT_PUBLIC_POSTHOG_ENABLED=true`
- `NEXT_PUBLIC_POSTHOG_KEY=<project_key>`
- `NEXT_PUBLIC_POSTHOG_HOST=https://us.i.posthog.com`

### C. 本地/预发验证清单
1. 登录后在 Workbench/Profile 操作，确认 PostHog 实时事件出现。
2. 确认 `distinct_id` 为业务 `userId`。
3. 触发 `generateShareLinkAction`，确认 `CUSTOMIZE_SHARE_CLICKED`。
4. 触发 customize 入队成功，确认 `CUSTOMIZE_SESSION_STARTED`。
5. 检查 replay：
- Workbench/Profile 有会话
- 公共分享页无会话
6. 检查隐私：输入内容、简历正文不可见（被 mask/block）。

### D. 数据质量阈值（上线门槛）
1. Outbox flush 成功率 >= 99%（排除外部故障窗口）。
2. Analytics 无明文邮箱、无完整 referrer URL。
3. 7 条漏斗在预发均能出数。

## 7.2 上线后（如何用到增长）

### Week 1（打地基）
1. 建好 7 条漏斗。
2. 建 3 个告警：
- `MATCH_FAILED` 突增
- `WORKER_PROVIDER_NOT_CONFIGURED > 0`
- `TASK_BACKPRESSURED` 比率超阈值
3. 每日 10 分钟看失败告警和漏斗突变。

### Week 2（用 Replay 找真问题）
1. 每周固定 15-30 分钟回放抽样（新用户 + 失败用户）。
2. 将回放问题映射到漏斗断点（不是只看“有趣录屏”）。
3. 形成“问题列表 -> 修复优先级”。

### Week 3（Survey 验证假设）
1. 选择一个漏斗断点触发调研（例如 customize 失败后）。
2. 只发 1-2 个短问题（避免低回复率）。
3. 把调研反馈回写到版本迭代说明。

## 7.3 最小产品增长闭环（可执行模板）
1. 发现：Funnel 显示 Step2 转化下降。
2. 定位：Replay 看到用户在某交互卡住。
3. 验证：Survey 询问“卡住原因/预期”。
4. 改版：上线小改动（按钮文案/交互节奏/容错提示）。
5. 评估：对比改版前后同漏斗转化与失败率。

## 7.4 团队节奏模板
1. Daily（10 分钟）
- 看告警 + 关键失败事件趋势。

2. Weekly（30-45 分钟）
- 复盘 7 漏斗
- 回放抽样
- 决定下周一个增长实验点

3. Monthly（60 分钟）
- 清理无价值埋点
- 调整采样/录屏范围
- 复审隐私策略与成本

## 9. Analytics v3.1（裂变 + Landing 归因）最小落地清单

### 9.1 本轮新增能力（已落地）
1. 新增 `LANDING_ATTRIBUTED` 事件，用于记录 Landing 首触/末触归因。
2. `SERVICE_CREATED` 增加归因快照字段（`acq*`），实现“来源 -> 业务创建”闭环。
3. 分享链路增强：
- `CUSTOMIZE_SHARE_CLICKED` 增加 `shareMethod`（`generate_link/copy_link/preview_open/renew_link`）
- `RESUME_SHARE_VIEW` 增加 `utm_content/utm_term`
- `RESUME_SHARE_CTA_CLICK` 增加 `utm_*` 全量参数
4. 分享链接默认附带渠道参数（`utm_source/medium/campaign/content + src + share_id`）。
5. Landing 页新增前端归因 Tracker：
- 写入 first-touch / last-touch cookie
- 每个 session 上报一次 `LANDING_ATTRIBUTED`

### 9.2 归因字段口径（统一口径，避免报表漂移）
1. `touch`
- `first`：用户首次触达来源
- `last`：本次 session 最近一次来源
2. `channel`（系统自动归类）
- `resume_referral` / `paid` / `organic_search` / `social` / `referral` / `direct` / `unknown`
3. `utm_*`
- `utm_source`：平台来源（如 `resume_share` / `google_ads` / `x`）
- `utm_medium`：流量介质（如 `referral` / `cpc` / `social`）
- `utm_campaign`：活动名（如 `resume_share_q2`）
- `utm_content`：素材/链接位（本项目建议默认放 `shareId`）
- `utm_term`：关键词（搜索投放场景可选）

## 10. 上线执行手册（你可以按这个顺序直接做）

### 10.1 上线前（T-3 ~ T-1）
1. 统一命名规则（必须先定）
- `utm_source`: 小写蛇形，如 `resume_share`, `google_ads`, `newsletter`
- `utm_medium`: 固定枚举，如 `referral`, `cpc`, `organic`, `social`
- `utm_campaign`: `yyyy_mm_topic`, 如 `2026_03_share_launch`
2. 准备渠道链接模板
- 简历分享链接（系统默认）：
  `/{locale}/r/{shareKey}?utm_source=resume_share&utm_medium=referral&utm_campaign=resume_share&utm_content={shareKey}&share_id={shareKey}&src=share`
- 投放链接：
  `/{locale}?utm_source=google_ads&utm_medium=cpc&utm_campaign=2026_03_core_cv&utm_term={keyword}`
3. 在 PostHog 建立 2 个核心漏斗
- 裂变漏斗：`CUSTOMIZE_SHARE_CLICKED -> RESUME_SHARE_VIEW -> RESUME_SHARE_CTA_CLICK -> SERVICE_CREATED`
- Landing 转化漏斗：`LANDING_ATTRIBUTED -> SERVICE_CREATED`
4. 建 3 个告警
- `MATCH_FAILED` 异常升高
- `TASK_BACKPRESSURED` 持续升高
- `LANDING_ATTRIBUTED` 正常但 `SERVICE_CREATED` 转化突降

### 10.2 上线当天（T0）
1. 用 3 条真实链接验证
- 分享链接
- 投放链接
- 无 UTM 直达链接
2. 检查事件字段完整性
- `LANDING_ATTRIBUTED` 是否带 `touch/channel/utm_*`
- `SERVICE_CREATED` 是否带 `acq*`
3. 检查归因一致性
- 同一浏览器从分享页进入 Landing 后创建 service，`acqChannel` 应为 `resume_referral`

### 10.3 上线后第 1 周（T+1 ~ T+7）
1. 每天看一次：
- `LANDING_ATTRIBUTED` 总量
- `SERVICE_CREATED` 总量
- 裂变漏斗各步转化率
2. 如果 `RESUME_SHARE_VIEW -> RESUME_SHARE_CTA_CLICK` 低：
- 优先改分享页 CTA 文案和按钮位置
3. 如果 `LANDING_ATTRIBUTED -> SERVICE_CREATED` 低：
- 优先改 Landing Hero 信息密度和首屏行动按钮

## 11. 长线运营：投放与监控怎么做

### 11.1 投放策略（低成本优先）
1. 先做“单变量投放”
- 每次只改一个维度（渠道或素材或文案），避免结果不可解释。
2. 每周只保留 Top 2 渠道
- 用 `SERVICE_CREATED` 成本和转化率决策，砍掉低效来源。
3. 分享裂变单独看
- 将 `resume_share` 作为独立渠道核算，不和广告混算。

### 11.2 核心监控指标（建议固定看板）
1. 裂变效率
- `share_to_view_rate = RESUME_SHARE_VIEW / CUSTOMIZE_SHARE_CLICKED`
- `view_to_cta_rate = RESUME_SHARE_CTA_CLICK / RESUME_SHARE_VIEW`
- `cta_to_service_rate = SERVICE_CREATED(resume_referral) / RESUME_SHARE_CTA_CLICK`
2. Landing 效率
- `landing_to_service_rate = SERVICE_CREATED / LANDING_ATTRIBUTED`
- 分渠道看：`by acqChannel, utm_source, utm_campaign`
3. 质量指标
- `MATCH_FAILED rate`
- `CUSTOMIZE_GENERATED success rate`
- `INTERVIEW_COMPLETED rate`

### 11.3 周会复盘模板（可直接照抄）
1. 本周增量来源（渠道/活动）
2. 漏斗断点（哪一步掉得最多）
3. 录屏抽样结论（Top 3 真实问题）
4. 下周仅做 1-2 个改动（明确预期提升指标）

### 11.4 迭代优先级规则
1. 先修“高流量断点”再修“低流量体验问题”。
2. 先做可逆的小改动（文案/顺序/按钮）再做大重构。
3. 每个改动必须绑定一个可量化指标（否则不进入本周）。

## 12. 快速排查 SQL（DB 真理源）

### 12.1 渠道到 Service 创建（近 7 天）

```sql
select
  payload->>'acqChannel' as acq_channel,
  payload->>'acqSource'  as acq_source,
  payload->>'acqCampaign' as acq_campaign,
  count(*) as service_created
from analytics_events
where event_name = 'SERVICE_CREATED'
  and created_at >= now() - interval '7 day'
group by 1,2,3
order by service_created desc;
```

### 12.2 裂变漏斗第一跳（分享行为质量）

```sql
select
  payload->>'shareMethod' as share_method,
  count(*) as cnt
from analytics_events
where event_name = 'CUSTOMIZE_SHARE_CLICKED'
  and created_at >= now() - interval '7 day'
group by 1
order by cnt desc;
```

### 12.3 Landing 归因分布

```sql
select
  payload->>'touch' as touch,
  payload->>'channel' as channel,
  payload->>'utm_source' as utm_source,
  count(*) as cnt
from analytics_events
where event_name = 'LANDING_ATTRIBUTED'
  and created_at >= now() - interval '7 day'
group by 1,2,3
order by cnt desc;
```

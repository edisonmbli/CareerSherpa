# CareerShaper MVP 站内反馈闭环方案

## Summary
- 你的真实需求不是“客服系统”，而是一个 `MVP 级 founder inbox`：让铁杆用户在当前使用场景里顺手反馈，同时系统自动补齐 `userId / serviceId / 当前模块 / 时间 / 路由 / 状态 / 回放线索 / 错误线索`，让你能快速判断是 `bug`、`需求` 还是 `认知卡点`。
- 社区成熟做法的共识是：`站内原位反馈入口 + 自动上下文 + Session Replay/错误追踪 + 定点微调研 + 异步分发到 Slack/邮件/工单`，而不是单靠邮箱或单独客服页。
- 对你当前栈，推荐主方案是：`PostHog + Sentry + 轻量自建反馈入口 + Slack/邮箱收件 + QStash 异步投递`。不新建主业务 DB 表，不引入重型客服平台。
- `PostHog Surveys` 可以用，但定位应是“定点提问/假设验证”，不是主反馈入口。`Sentry User Feedback` 适合“报错后补充信息”，也不适合承接通用需求池。

## Key Changes
- 在现有右下角浮动操作区新增统一反馈入口，优先接在 [components/app/ServiceDisplay.tsx](/Users/edisonmbli/Projects/CareerShaper/components/app/ServiceDisplay.tsx)、Profile、Billing 的同类固定入口；桌面和移动端都保持“一步打开，不跳出当前业务场景”。
- 反馈面板只保留 3 个主类型：`报告问题`、`提出需求`、`这里让我困惑`。表单只要一个主文本框，加一个可选“允许邮件回复我”开关；不要让用户自己填写技术细节。
- 提交时自动附带上下文：`userId`、`plan/tier`、`locale`、`route`、`serviceId`、当前 tab（match/customize/interview/profile/billing）、当前任务状态、时间戳、浏览器/视口、PostHog `distinct_id`、最近一次 Sentry `eventId`。默认不上传原始简历/JD 正文，只传 ID 和派生元数据。
- 反馈 API 设计为 `POST /api/feedback`，服务端只做校验和异步投递；使用现有队列基础设施而不是写主业务库。投递目标先定为 `Slack/邮箱`，高优先级 bug 再人工转 `Linear`。
- 每次反馈同时打产品事件到 PostHog，例如 `FEEDBACK_OPENED`、`FEEDBACK_SUBMITTED`、`FEEDBACK_SUBMITTED_SUCCESS`，属性包含 `feedback_type`、`surface`、`service_id_present`、`had_sentry_event`。这样你能用时间点和 person timeline 去对齐 replay。
- 对异常态单独加“报案快捷入口”：在 [components/ui/error-display.tsx](/Users/edisonmbli/Projects/CareerShaper/components/ui/error-display.tsx) 把当前 `mailto:` 兜底升级为“带 eventId 的问题反馈”。这条链路优先接 Sentry 线索。
- 保留现有 [components/analytics/PostHogProvider.tsx](/Users/edisonmbli/Projects/CareerShaper/components/analytics/PostHogProvider.tsx) 的 replay 能力，只新增反馈相关属性注册与事件采集；不扩大录屏范围，不采正文。

## Tool Decisions
- 主栈决策：`PostHog` 负责行为、person、replay、定点 survey；`Sentry` 负责异常与 error-linked user feedback；自建轻反馈面板负责统一入口；`Slack/邮箱` 做 founder inbox。这个组合对你最省钱、最少集成、也最符合现有仓库事实。
- `PostHog Surveys` 的使用边界：只在高摩擦节点触发 1 个短问题，例如“定制失败后”“导出后”“连续重试后”；不要让 Survey 承担常驻反馈入口。[PostHog Surveys](https://posthog.com/surveys) 官方定位就是收集用户反馈与研究，而不是替代完整工单流。
- `Sentry User Feedback` 的使用边界：只用于“刚发生异常/报错”的场景，因为它天然适合和异常事件绑定。[Sentry JavaScript User Feedback](https://docs.sentry.io/platforms/javascript/guides/react/user-feedback/) 支持直接提交用户反馈，但不适合作为通用需求池。
- 未来如需开源/低成本替代：`Formbricks` 适合自托管 survey / in-app microsurvey，官方定价页明确自托管免费、Cloud 有低价档；但它更偏“调研系统”，不是 bug triage 主入口。[Formbricks Pricing](https://formbricks.com/pricing)
- 未来如需公开投票板/路线图：再评估 `Featurebase` 这类产品；它适合“公开收集 + 投票 + changelog + portal”，但对你当前私密 founder inbox 阶段偏重。[Featurebase Pricing](https://featurebase.app/pricing)
- 明确不推荐：继续把 `support@...` 作为主入口；邮件只能保留为兜底通道，因为它无法稳定附带上下文，也不利于和 replay / error / person 关联。

## Test Plan
- 从 `match / customize / interview / profile / billing` 进入反馈，提交后 payload 都能带上正确 `surface`、`route`、`serviceId`、`userId`、当前状态。
- 提交反馈不会阻塞主业务交互，接口超时或第三方 webhook 失败时，前端显示“已收到/稍后重试”而不影响主流程。
- bug 反馈在存在异常时能带上 `Sentry eventId`；需求反馈不强依赖 Sentry。
- PostHog 中可以按 `feedback_type`、`surface`、提交时间回看 person timeline / replay；运营上能从 Slack 消息直接跳到用户与场景。
- 隐私验证必须通过：默认不发送原始简历正文、JD 正文、输入框明文；只发送 ID、状态、元数据和用户主动填写的反馈文本。
- Survey 只在指定事件或路径触发，不全站乱弹；回复率和噪音受控。
- FAQ 演进机制固定为每周人工归档：重复问题先沉淀成模块内帮助文案，再决定是否升级为静态 FAQ 页面。

## Assumptions
- 默认按你刚确认的方向执行：`私密收集`，处理落点先用 `Slack/邮箱`，不做公开投票板。
- 默认不新增主业务 DB 写入；反馈数据以 `异步 webhook / 队列 / 第三方系统` 为主，主站只承担轻量 API 编排。
- 默认先不上截图上传；如果后续发现“还原现场”仍然困难，再加可选截图或屏幕区域截图。
- 默认 FAQ 不是先做独立帮助中心，而是先做“场景内 FAQ / 帮助提示 / 空状态文案”。
- 结论性判断：`PostHog Surveys 能用于这个场景，但只能做辅助手段；主反馈入口必须是常驻、原位、自动带上下文的反馈面板。`

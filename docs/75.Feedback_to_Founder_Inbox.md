# Feedback to Founder Inbox

## 1. 目标

本方案解决的不是“客服工单系统”，而是 `MVP 阶段 founder inbox`：

1. 用户在当前业务场景里顺手反馈，不需要跳到独立客服页面。
2. 系统自动附带 `userId / serviceId / 当前模块 / 状态 / 时间 / PostHog distinct_id / Sentry eventId`。
3. 反馈先写入 `feedback_inbox`，它是 founder inbox 的唯一事实源。
4. `Slack` 是异步通知通道，由创始人快速判断：`bug`、`需求`、`认知卡点`。
5. 高频共性问题再沉淀为 FAQ 或产品内说明，避免重复解释。

当前代码实现包含：

- 站内反馈弹层入口
- `/api/feedback` 提交接口
- `feedback_inbox` 持久化
- Slack webhook / bot thread 投递
- QStash 可选异步投递
- PostHog 埋点：`FOUNDER_FEEDBACK_OPENED`、`FOUNDER_FEEDBACK_SUBMITTED`、`FOUNDER_FEEDBACK_SUBMIT_FAILED`

## 2. 当前接入点

已落地入口：

1. `workbench` 页面
2. `profile` 页面
3. `billing` 页面（通过 profile tab）

自动附带上下文：

1. `locale`
2. `surface`（workbench / profile / billing）
3. `tab`
4. `serviceId`
5. `status`
6. `tier / queueType`
7. `currentUrl / pathname / title`
8. `Sentry.lastEventId()`
9. `posthog.get_distinct_id()`
10. `posthog.get_session_id()`
11. `posthog.get_session_replay_url({ withTimestamp: true, timestampLookBack: 30 })`
12. `viewport / timezone / userAgent`
13. 一小组派生 extra fields，例如 `matchReady / interviewReady / connected / serviceNetCoinDelta`
14. 若能稳定映射到当前任务，还会附带 `taskId / taskTemplateId / taskTierHint`
15. 投递阶段会额外 enrich：
    - `coin_transactions`：判断该任务是否真实扣费、扣费状态、扣费值
    - `llm_usage_logs`：判断该任务实际使用了哪个 `provider / modelId`

默认不发送的内容：

1. 原始简历全文
2. 原始 JD 全文
3. 输入框明文内容
4. 页面截图

这是一条明确的隐私边界。MVP 阶段先靠 `ID + 状态 + 回放线索` 定位问题，不做高风险正文外发。

## 2.1 关于 DB enrich 的判断

新增的 founder inbox enrich 只发生在服务端 delivery / dispatch 阶段，不发生在用户点击提交之前。

这样设计的原因：

1. 反馈是低频路径，不是主业务高并发链路。
2. `coin_transactions` 和 `llm_usage_logs` 都有可用索引，查询维度也足够窄。
3. 若使用 QStash，这类 enrich 在异步 dispatch 里完成，不阻塞用户页面。
4. enrich 失败只影响 founder 看见的上下文丰富度，不影响业务结果本身。

查询原则：

1. 有 `taskId` 时，优先按 `userId + serviceId + taskId` 精确命中扣费记录。
2. 没有精确命中时，退回 `serviceId + taskTemplateId`。
3. 模型查询按 `userId + serviceId + taskTemplateId` 取对应时间窗口内最近的 `llm_usage_logs`。

注意：

1. `taskTierHint` 是 UI 层 hint，不等于真实计费真值。
2. 真实 paid/free 判断以 `coin_transactions` 命中结果为准。
3. 真实模型判断以 `llm_usage_logs.provider + modelId` 为准。

## 3. 环境变量

至少配置一种投递通道。

### 3.1 Slack webhook

```bash
FEEDBACK_SLACK_WEBHOOK_URL=https://hooks.slack.com/services/xxx/yyy/zzz
FEEDBACK_SLACK_MENTION=@yourname
FEEDBACK_SLACK_BOT_TOKEN=xoxb-...
FEEDBACK_SLACK_CHANNEL_ID=C0123456789
```

说明： 

1. `FEEDBACK_SLACK_WEBHOOK_URL` 是主推荐方案，最省事。
2. `FEEDBACK_SLACK_MENTION` 可选，用于在消息顶部主动提醒你。
3. 如果要启用 `精简版主消息 + 详细版 thread reply`，还需要 `FEEDBACK_SLACK_BOT_TOKEN + FEEDBACK_SLACK_CHANNEL_ID`。
4. 如果只配了 webhook，没有 bot token，则会自动降级为单消息模式。
5. 如果只想走 Slack，不必配置 Resend。
6. Slack bot 至少需要 `chat:write` scope；若频道策略需要，也可能还要 `chat:write.public`。

### 3.3 QStash 可选异步投递

```bash
QSTASH_TOKEN=...
QSTASH_CURRENT_SIGNING_KEY=...
QSTASH_NEXT_SIGNING_KEY=...
NEXT_PUBLIC_APP_BASE_URL=https://your-app.com
```

说明：

1. 配齐 QStash 后，`/api/feedback` 会先同步写入 `feedback_inbox`，再入队，由 `/api/feedback/dispatch` 异步投递。
2. 不配 QStash 时，接口仍会先同步写入 `feedback_inbox`，然后直接尝试同一套 dispatch 逻辑。
3. 对用户来说，提交成功的定义是“写入 `feedback_inbox` 成功”，不是 Slack 成功。
4. 生产环境推荐启用 QStash，原因是 webhook 超时不会阻塞用户主流程。
4. feedback 走的是独立 dispatch URL，而不是 LLM worker handler。
5. 你在 Slack 里看到的 `deliveryMode=qstash` 只表示传输通道，不表示它进入了某个业务 queue 名。
6. 当前 founder inbox 的 QStash retry 已提高到 `3`，更适合这类非核心、可接受延迟的异步通知。

### 3.4 QStash 与 LLM 任务是否会互相影响

当前实现中，feedback 和 LLM 任务共享的是同一个 QStash 项目基础设施，但不是同一个业务 endpoint。

区别在于：

1. feedback 固定投递到 `/api/feedback/dispatch`
2. LLM 任务投递到各自 worker / producer 对应链路

所以当前不会出现“feedback body 跑进 LLM worker 逻辑”这种错误路由问题。

真正共享的只有：

1. QStash 项目配额
2. 重试基础设施
3. 监控面板

MVP 阶段这是合理的。只有当 founder inbox 量显著增加，或你明确需要更强隔离时，再拆专用 token / project / flow control。

## 4. Slack 配置指南

推荐路径：`Slack App -> Incoming Webhooks -> Activate -> Add New Webhook to Workspace`

建议做法：

1. 单独新建频道，例如 `#founder-inbox` 或 `#product-feedback`。
2. 只放结构化反馈，不要和普通讨论混在一个频道。
3. 保持 webhook 仅用于这个功能，不和 deploy / cron / alert 混用。
4. 频道里手动维护三种 emoji 约定：
   - `:beetle:` = bug
   - `:sparkles:` = feature
   - `:thinking_face:` = confusion
5. 每周把高频反馈转成一页 FAQ 或一条产品改进任务，不要让频道无限堆积。

Slack 消息里已经包含：

1. 用户原始反馈
2. 用户 ID
3. 账户邮箱（若允许）
4. `surface / tab / status / serviceId`
5. `PostHog distinct_id`
6. `Sentry eventId`
7. 浏览器与视口信息
8. 若可回查成功，还会带上 `billingMode / billingStatus / debitDelta / provider / modelId`
9. 若 PostHog 配置齐全，还会带上：
   - `Open Person`
   - `Open Replay`

消息结构已升级为 `Block Kit V2`，详细定义见 [75_1.Feedback_Slack_Messages_Structure.md](/Users/edisonmbli/Projects/CareerShaper/docs/75_1.Feedback_Slack_Messages_Structure.md)。

当前推荐的 MVP Slack 发送策略：

1. 主消息：只保留 triage 必需信息
2. thread reply：放 `Identifiers / Billing / Page / Debug` 详情
3. 若 bot token 未配置，或 bot 发送失败，则自动退回到“单消息含完整详情”的 webhook 模式

### 4.1 PostHog deep link 配置

Replay deep link 由浏览器端 PostHog SDK 直接生成，因此只要：

1. `NEXT_PUBLIC_POSTHOG_ENABLED=true`
2. `NEXT_PUBLIC_POSTHOG_KEY`
3. `NEXT_PUBLIC_POSTHOG_HOST`
4. 当前页面在 `PostHogProvider` 范围内
5. 当前 scope 开启 session recording

就能把 `Open Replay` 放进 Slack。

Person deep link 需要额外配置一条 app 侧 URL 规则。推荐至少配置：

```bash
POSTHOG_APP_BASE_URL=https://us.posthog.com
POSTHOG_PROJECT_ID=123456
```

如果你的 PostHog workspace/person 路由不是默认格式，再额外配置：

```bash
POSTHOG_PERSON_URL_TEMPLATE=https://us.posthog.com/project/{projectId}/person/{distinctId}
```

可用占位符：

1. `{appBaseUrl}`
2. `{projectId}`
3. `{distinctId}`
4. `{sessionId}`
5. `{replayUrl}`

## 5. 持久化与投递状态

当前 founder inbox 的真实链路是：

1. 浏览器提交 `POST /api/feedback`
2. 服务端校验并同步写入 `public.feedback_inbox`
3. 若配置了 QStash，则仅把 `feedbackId + deliveryMode` 入队
4. `/api/feedback/dispatch` 根据 `feedbackId` 查库
5. 查库后 enrich：`coin_transactions / llm_usage_logs`
6. 投递 Slack
7. 回写投递状态

`feedback_inbox` 的状态字段：

1. `PENDING`
2. `PROCESSING`
3. `DELIVERED`
4. `FAILED`

核心原则：

1. `feedback_inbox` 是 single source of truth
2. Slack 只是通知视图，不是事实源
3. 只要写库成功，就视为用户反馈已被系统接收
4. Slack/QStash 故障只会影响通知，不会导致反馈数据丢失

## 6. PostHog 使用指引

### 6.1 反馈事件

本次新增了三个事件：

1. `FOUNDER_FEEDBACK_OPENED`
2. `FOUNDER_FEEDBACK_SUBMITTED`
3. `FOUNDER_FEEDBACK_SUBMIT_FAILED`

关键属性建议在 PostHog 里建立 breakdown：

1. `surface`
2. `tab`
3. `type`
4. `status`
5. `queued`
6. `serviceId`

### 6.2 如何回放定位问题

当你收到一条反馈时，推荐按这个顺序排查：

1. 先看 Slack/邮件里的 `type` 和用户原文。
2. 打开 PostHog，按 `distinct_id + 提交时间窗口` 找到对应 person timeline。
3. 如果是 workbench 反馈，再按 `serviceId` 回到业务数据页确认状态。
4. 如果用户提到的是错误或失败，优先看 `Sentry eventId` 对应的异常。
5. 最后才看 session replay，而不是先盲刷录屏。

### 6.3 Survey 的正确定位

`PostHog Surveys` 只做“定点提问”，不要替代常驻反馈入口。

适合触发 Survey 的场景：

1. `CUSTOMIZE_FAILED` 之后，问一句“你本来想完成什么？”
2. 导出后，问一句“这份结果最有帮助的是哪一段？”
3. 连续重试后，问一句“卡住点在哪里？”

不适合让 Survey 做的事：

1. 作为所有页面唯一反馈入口
2. 收集长文本 bug 报告
3. 承担 founder inbox 的主流程

原因很简单：Survey 是“你问用户”，反馈面板是“用户主动告诉你”。两者职责不同。

## 7. Sentry 使用指引

最佳实践：

1. `Sentry` 负责异常与错误堆栈。
2. 站内反馈面板负责通用 bug / feature / confusion。
3. 如果某个错误态页面已经拿到了明确的 `eventId`，优先把这个 `eventId` 一起带到 founder inbox。
4. 不要把 Sentry 当需求池，它不是干这个的。

建议你形成这个排查顺序：

1. 先看 Sentry 是否有异常聚类。
2. 再看 PostHog 是否有行为断点与 replay。
3. 最后结合用户原话判断是代码 bug、产品设计问题，还是文案认知问题。

## 8. 每日 / 每周运营节奏

### 8.1 Daily（10 分钟）

1. 打开 Slack `#founder-inbox`。
2. 先处理 `bug`，再处理 `confusion`，最后看 `feature`。
3. 对每条反馈只做一个动作：
   - 立刻修
   - 记录观察
   - 合并到已有问题
   - 暂时不做

### 8.2 Weekly（30-45 分钟）

1. 统计本周反馈数与分类占比。
2. 抽样看 5-10 条 replay。
3. 挑 3 条最高频问题做产品修复或文案修复。
4. 把重复 3 次以上的问题升级为 FAQ / inline help / empty-state 文案。

### 8.3 Monthly（45-60 分钟）

1. 清理无价值反馈类型或过长表单字段。
2. 复盘哪些上下文字段真的有帮助，删掉噪音字段。
3. 重新审视是否需要引入公开路线图或投票板。

## 9. FAQ 沉淀规则

前期不要先做大而全的帮助中心。

建议的沉淀顺序：

1. 先做模块内提示文案
2. 再做空状态说明
3. 再做结果页小 FAQ
4. 最后才做集中 FAQ 页面

判断一个问题是否该进入 FAQ：

1. 同类问题一周内重复出现 >= 3 次
2. 问题不是代码 bug，而是认知摩擦
3. 可以用一小段文案或 UI 提示解决

FAQ 的目标不是“堆内容”，而是“减少下次同类反馈”。

## 10. 隐私与边界

必须坚持：

1. 不外发简历全文和 JD 全文
2. 不默认截图
3. 不写主业务 DB
4. 不把 founder inbox 变成新的数据黑洞

如果未来确实需要截图：

1. 只做用户主动授权上传
2. 明确告知截图会发送到哪里
3. 先脱敏，再上传
4. 不要默认自动截屏

## 11. 故障排查

### 11.1 点击提交后提示未配置

说明没有配置任何有效投递通道。

检查：

1. `FEEDBACK_SLACK_WEBHOOK_URL`
2. `FEEDBACK_RESEND_*`

### 11.2 已配置 QStash，但没有实际投递

检查：

1. `QSTASH_TOKEN`
2. `QSTASH_CURRENT_SIGNING_KEY`
3. `QSTASH_NEXT_SIGNING_KEY`
4. `NEXT_PUBLIC_APP_BASE_URL`
5. 生产域名是否允许 `/api/feedback/dispatch`

### 11.3 Slack 收到，但上下文不全

检查：

1. 当前页面是否在 `PostHogProvider` 范围内
2. 用户是否登录
3. 页面是否传入了 `serviceId / tab / status`
4. 浏览器是否拦截了脚本或扩展导致 `posthog.get_distinct_id()` 为空

## 12. 未来升级路线

如果 founder inbox 跑顺了，再考虑这些升级：

1. 给高频问题自动打标签
2. 自动把重复反馈聚类成主题
3. 自动同步高优 bug 到 Linear
4. 给已解决问题生成 changelog 摘要
5. 把共性需求同步到公开 roadmap

升级顺序建议：

1. 先把 founder inbox 跑顺
2. 再加自动分类
3. 再加 FAQ 沉淀
4. 最后才考虑公开投票与路线图

不要倒过来。MVP 阶段最重要的是“快速收到真问题并快速处理”，不是先搭一个看起来很完整的反馈平台。
